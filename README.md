# Система постановки поручений и контроля за их исполнением — школа № 2090

Внутренний инструмент одной организации (школа № 2090) для постановки поручений и контроля их
исполнения — нечто среднее между таск-трекером и электронным документооборотом: **руководитель ставит
поручение → исполнитель отчитывается о выполнении → постановщик контролирует и закрывает**.
Приоритет определяется **только сроком**: «что у тебя горит, то и приоритет».

Сервис самодостаточен, разворачивается одной командой `docker compose up`, раздаётся бесплатно и
открыто (лицензия MIT) для self-host. Язык интерфейса — русский.

> Это **не** Jira и **не** Kanban-доска: задачи показываются таблицей, ролей-аккаунтов нет —
> роли (постановщик / исполнитель / наблюдатель) назначаются **на каждую конкретную задачу**.

## Возможности

- **Доступ по белому списку e-mail.** Зарегистрироваться может только тот, чью почту администратор
  внёс в реестр. Логин = e-mail, восстановление пароля по ссылке.
- **Три ипостаси** (переключаются в любой момент): «Я постановщик», «Я исполнитель», «Я наблюдатель».
- **Задача:** сквозной номер + 6-значный код для поиска, название, описание, вложения (файлы любого
  типа и/или ссылки), срок («дата+время» или «только дата»), один исполнитель, до 5 наблюдателей,
  статус (В работе / Выполнена / Отменена), отдельный авто-флаг **«Просрочена»**, отчёт исполнителя и
  отметка готовности.
- **Серверная проверка прав на каждом действии** — ограничения нельзя обойти через API.
- **Уведомления** по e-mail (обязательно) и MAX (best-effort): график из 4 событий
  (постановка → за сутки → в день срока → ежедневно после срока) с журналом идемпотентности.
- **Выгрузка списка под печать** (готовая HTML-страница).
- **Адаптивная вёрстка** — корректно работает с телефона.

## Архитектура

Модульный монолит + тонкий воркер (минимум движущихся частей для школьного self-host):

| Контейнер  | Что это                                                          |
|------------|------------------------------------------------------------------|
| `db`       | PostgreSQL 16 (том `pgdata`)                                      |
| `backend`  | FastAPI / uvicorn — REST API, авторизация по JWT                  |
| `worker`   | тот же образ — APScheduler: флаг просрочки + суточная рассылка    |
| `web`      | Caddy — авто-HTTPS, раздаёт собранный SPA и проксирует `/api`     |

Один веб-контейнер (Caddy) и терминирует TLS, и раздаёт статику фронта, и проксирует API —
отдельный nginx не нужен. Вложения хранятся в файловом томе `attachments` и отдаются **только**
через эндпойнт с проверкой прав.

**Режим «3 контейнера».** Для совсем скромного VPS планировщик можно встроить в backend-процесс:
задайте `SCHEDULER_IN_PROCESS=true` в `.env` и запускайте без `worker`:
`docker compose up -d db backend web`. По умолчанию планировщик — в отдельном контейнере (изоляция
отказов); идемпотентность рассылки в обоих режимах гарантируется журналом `notification_log`.

Заложены лимиты ресурсов (`mem_limit`/`cpus`) и ротация логов (`json-file`, 10 МБ × 3) — чтобы один
сервис не съел память и логи не забили диск VPS.

## Требования

- Docker и Docker Compose (v2). Больше ничего ставить не нужно.

## Быстрый старт

```bash
git clone <repo-url> task-hub-2090
cd task-hub-2090
cp .env.example .env
# отредактируйте .env: задайте POSTGRES_PASSWORD, JWT_SECRET, SMTP_*, ADMIN_EMAIL, ADMIN_PASSWORD
docker compose up -d --build
```

После старта:

- интерфейс: ваш домен (через Caddy с авто-HTTPS) или `http://<сервер>` при `SITE_ADDRESS=:80`;
- документация API (Swagger): `<адрес>/api/v1/docs`.

При первом запуске контейнер `backend` автоматически применяет миграции и (если `RUN_SEED=true`)
создаёт **первичного администратора** из `ADMIN_EMAIL` / `ADMIN_PASSWORD` и вносит его e-mail в реестр.

### Дальнейшее наполнение

1. Войдите администратором → раздел **«Администрирование»** → внесите e-mail сотрудников в реестр
   (типовое первичное наполнение — все работники школы).
2. Сотрудник с внесённым e-mail сам проходит регистрацию (задаёт пароль) и входит.
3. Любой зарегистрированный пользователь может поставить задачу любому.

## Переменные окружения

Полный список — в [`.env.example`](.env.example). Ключевые группы:

| Группа              | Переменные                                                          |
|---------------------|--------------------------------------------------------------------|
| База данных         | `POSTGRES_USER/PASSWORD/DB`, `DATABASE_URL`                         |
| Секреты сессии      | `JWT_SECRET`, `JWT_ACCESS_TTL_MIN`, `REFRESH_TTL_DAYS`, `PASSWORD_RESET_TTL_MIN` |
| SMTP (обязательно)  | `SMTP_HOST/PORT/USER/PASSWORD/TLS/FROM`                            |
| MAX (опционально)   | `MAX_ENABLED`, `MAX_API_BASE`, `MAX_BOT_TOKEN`, `MAX_BLOCKED_EXT`   |
| Вложения / лимиты   | `ATTACHMENTS_DIR`, `MAX_FILE_SIZE_MB`, `MAX_FILES_PER_TASK`, …      |
| Таймзона/расписание | `TZ`, `NOTIFY_TIME`, `OVERDUE_SWEEP_MINUTES`                        |
| Адрес/доступ        | `BASE_URL`, `CORS_ORIGINS`, `SITE_ADDRESS`                         |
| Администратор/сид   | `ADMIN_EMAIL`, `ADMIN_PASSWORD`, `RUN_SEED`, `SEED_REGISTRY_FILE`   |

> **Безопасность:** обязательно смените `JWT_SECRET` и пароли в проде. Реальный `.env` не коммитится
> (`.gitignore`); в репозитории — только `.env.example`.

## Настройка SMTP и MAX

- **E-mail — обязательный канал.** Укажите параметры вашего SMTP-сервера (`SMTP_*`). Это адрес по
  умолчанию для всех уведомлений и ссылок восстановления пароля.
- **MAX — по возможности (best-effort).** Включается `MAX_ENABLED=true` + `MAX_API_BASE` и
  `MAX_BOT_TOKEN`. Привязка пользователя к MAX (`maxContact`) задаётся администратором в реестре.
  Если привязки нет — уведомление уходит только по e-mail. Проблемные для MAX вложения дублируются
  ссылкой и почтой.

## Развёртывание на школьном сервере с динамическим IP

Адрес задаётся через `BASE_URL` / `SITE_ADDRESS` — это единственная точка правки при смене адреса.

| Вариант | Когда                                  | Что делать                                                            |
|---------|----------------------------------------|----------------------------------------------------------------------|
| 1       | Провайдер даёт статический IP          | Заказать статический IP, пробросить порт; `SITE_ADDRESS` = домен/IP   |
| 2       | Школьный сервер недоступен извне       | Развернуть тот же `docker compose` на дешёвом VPS со статическим IP   |
| 3       | Нет белого IP и нет VPS                | Туннель (например, Cloudflare Tunnel) с публичного домена на сервер   |

Для автоматического HTTPS укажите в `.env` доменное имя в `SITE_ADDRESS` (например
`SITE_ADDRESS=school2090.example.org`) — Caddy сам выпустит и продлит сертификат Let's Encrypt.
Для HTTP-демо без домена оставьте `SITE_ADDRESS=:80`.

Если у школы уже есть свой reverse-proxy/туннель, оставьте `SITE_ADDRESS=:80` — контейнер `web`
будет отдавать сайт по HTTP на порту 80, направьте на него ваш внешний прокси.

## Бэкапы и обслуживание (важно для одного VPS)

Один сервер — одна точка отказа по диску, поэтому настройте бэкапы. Скрипт
[`infra/backup.sh`](infra/backup.sh) делает дамп БД (`pg_dump`) и архив вложений с ротацией:

```bash
./infra/backup.sh                      # разовый запуск (копии → ./backups)
# по расписанию — в crontab хоста, каждый день в 03:30:
# 30 3 * * * cd /opt/task-hub-2090 && ./infra/backup.sh >> /var/log/taskhub-backup.log 2>&1
```

Восстановление — команды приведены в комментариях внутри скрипта. Рекомендуется копировать
`./backups` на внешнее хранилище (другой сервер/облако).

## Локальная разработка

**Backend** (нужен Python 3.12):

```bash
cd backend
python -m venv .venv && .venv/bin/pip install -e ".[dev]"
# тест-БД в Docker:
docker compose -f ../docker-compose.test.yml up -d
export TEST_DATABASE_URL=postgresql+asyncpg://postgres:test@localhost:5433/test
.venv/bin/pytest            # автотесты
.venv/bin/uvicorn app.main:app --reload   # API на :8000
```

**Frontend** (нужен Node 20+):

```bash
cd frontend
npm install
npm run dev      # Vite на :5173, проксирует /api на :8000
npm run build    # production-сборка
npm test         # vitest
```

## Тесты

- **Backend:** `pytest` — матрица прав (позитив/негатив, 404-вместо-403), сквозная нумерация без дыр,
  уникальность 6-значного кода с повтором при коллизии, флаг просрочки в обоих режимах, график из 4
  уведомлений (получатели, тайминг, краевые случаи, идемпотентность, best-effort MAX), удаление
  пользователя с архивом и блокировкой входа, дословная фраза отказа в регистрации.
- **Frontend:** `vitest` — критичные компоненты (бейджи статуса/просрочки, расчёт просрочки, дословная
  фраза отказа).
- Обратимость миграций: `alembic upgrade head && alembic downgrade base && alembic upgrade head`.

```
│   └── tests/                  # unit + integration

│   └── src/                    # app / features / shared

    ├── caddy/Caddyfile         # веб-контейнер: TLS + статика SPA + проксирование /api

```
