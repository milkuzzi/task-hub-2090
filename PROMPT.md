## 1. РОЛЬ И КОНТЕКСТ

Построй адаптивное веб-приложение **«Система постановки поручений и контроля за их исполнением»** для одной организации — школы № 2090. Это **внутренний инструмент одной организации**, а не публичный сервис и не SaaS для разных компаний.

Назначение системы — **упрощённый документооборот**, нечто среднее между Jira (инструментом работы над проектом) и электронным документооборотом: руководитель ставит поручение → исполнитель отчитывается о выполнении → постановщик контролирует и закрывает. Цель заказчика дословно: «Меня очень сильно волнует понимание того, кто что в данный момент делает, и люди должны понимать, когда у них дедлайн».

Ключевые принципы:
- Приоритет определяется **только сроком** («что у тебя горит, то и приоритет») — отдельного поля приоритета нет.
- Одна организация, один контур. Сценарий первичного наполнения: администратор загружает в реестр **всех работников школы** (см. разделы 3, 13).
- Задачи **плоские** (без проектов, иерархии, декомпозиции). Группировка по проектам — только будущее расширение (см. раздел 12), сейчас НЕ реализуется, но модель данных должна оставлять возможность добавить поле «проект» без переписывания.
- Срок-ориентир — демонстрация другим школам и работодателям в середине июня; система раздаётся бесплатно/открыто, выкладывается на GitHub для self-host.
- Язык интерфейса — русский.

## 2. ЧЕМ ЭТО НЕ ЯВЛЯЕТСЯ (анти-рамки — НЕ реализовывать)

Заказчик прямо отверг перечисленное ниже. Не добавляй ничего из этого даже «для удобства»:

- **НЕ Jira и НЕ Kanban-доска.** Никаких колонок, досок, перетаскивания задач между колонками, backlog, пользовательских колонок, перемещения карточек по статусам. Задачи показываются **таблицей**, а не доской.
- **НЕТ приоритета** задач (поле отвергнуто: приоритет = срок).
- **НЕТ сложности** задачи (поле complexity отвергнуто).
- **НЕТ спринтов**, итераций, оценок трудозатрат.
- **НЕТ мультитенантности / мультикомпанийности.** Конкретно отвергнуто заказчиком: регистрация «администратора компании» с вводом названия компании, отдельные страницы под разные компании, использование сервиса в нескольких организациях одновременно. Строго одна школа.
- **НЕТ чата / обсуждений / тредов комментариев** внутри задачи. Нужен контроль исполнения, а не переписка. Исполнитель только добавляет отчёт о выполнении (текст и/или вложение).
- **НЕТ глобального назначения ролей** при регистрации. Роли (постановщик/исполнитель/наблюдатель) назначаются **на каждую конкретную задачу**, а не закрепляются за аккаунтом.
- **НЕТ коллективной ответственности.** Исполнитель всегда ровно один.
- **Просрочка — НЕ статус** и не заменяет статус. Это отдельный независимый автоматический флаг.
- **НЕТ push-уведомлений и SMS** сейчас (только архитектурный задел на будущее).
- **НЕТ сущности «проект» / группировки / дерева задач** сейчас (только будущее расширение).
- **НЕТ отдельного логина**: идентификатор пользователя — это e-mail.
- **НЕ показывать** пользователю без доступа никаких контактов/инструкций «как добавиться» — только фиксированная фраза (см. раздел 3).

## 3. ДОСТУП И АУТЕНТИФИКАЦИЯ

1. **Белый список (реестр) e-mail.** Пользоваться сервисом и регистрироваться может только тот, чья почта заранее внесена администратором в реестр. Сервис рассчитан на узкий круг (защита от наплыва посторонних). Типовое первичное наполнение реестра — все сотрудники школы.
2. **Логика «сначала доступ, потом самостоятельная регистрация»:** администратор вносит e-mail в реестр → владелец этой почты сам проходит регистрацию (задаёт пароль) → затем входит. Это именно регистрация, а не предсозданный аккаунт.
3. **Регистрация:**
   - Пользователь вводит e-mail.
   - Если e-mail **отсутствует** в реестре → показать ровно текст: **«Извините, у вас нет доступа к сервису.»** — и не пускать дальше. Никаких контактов, переадресации почты или подсказок «как получить доступ» не показывать.
   - Если e-mail **есть** в реестре → пользователь задаёт пароль и завершает регистрацию.
4. **Логин = e-mail.** Пароль привязан к почте. Отдельного поля «логин» нет.
5. **Вход:** e-mail + пароль.
6. **Восстановление/сброс пароля** обязателен — через e-mail (ссылка/токен со сроком жизни). Реализуй именно восстановление пароля, а не повторную регистрацию.
7. На странице входа/регистрации — **заметное предупреждение**: «Регистрация возможна только для адресов из утверждённого списка».
8. **Администратор** (директор) — отдельная привилегия в системе. Один и тот же человек может быть и обычным пользователем (ставить/исполнять/наблюдать задачи), и иметь доступ к админ-панели. Первичный администратор создаётся сид-скриптом/переменными окружения при развёртывании.
9. **Админ-панель — ведение реестра почт:** добавить, обновить, удалить запись прямо в интерфейсе. Список можно менять в любой момент.
10. **Удаление пользователя из реестра удаляет весь его архив** (директор дословно: «если я кого-то удаляю, то весь его архив тоже удаляется»). Удаляются все задачи, где он постановщик, и все его данные/вложения. Это **осознанное действие через отдельную кнопку с обязательным подтверждением** (диалог «удалить пользователя и весь его архив?»).
    - **Проектное решение (требует подтверждения, раздел 13):** объём «архива» директор не уточнял. Что делать со следами удаляемого в **чужих** задачах (где он был лишь исполнителем или наблюдателем) — заказчиком не задано. По умолчанию предлагается **не удалять чужие задачи целиком**, а помечать его участие как «пользователь удалён» (чтобы не ломать целостность чужих задач). Согласовать с заказчиком: снять / обезличить / пометить.
11. Если пользователя удалили из реестра — он теряет доступ (вход блокируется).
12. Серверная проверка доступа на каждый запрос: токен/сессия проверяется, удалённый из реестра пользователь не проходит.

## 4. РОЛИ (назначаются НА КАЖДУЮ ЗАДАЧУ, а не глобально)

Любой зарегистрированный пользователь может в разных задачах выступать в любой из трёх ролей. «Каждый может поставить задачу любому».

- **Постановщик** — создаёт задачу и полностью ею управляет. Заполняется автоматически тем, кто создал задачу.
- **Исполнитель (ответственный)** — **ровно один** на задачу. Выбирается из списка зарегистрированных пользователей (не вводится вручную — ему шлются уведомления на почту/MAX). Коллективной ответственности нет; как исполнитель организует работу со своими людьми — его дело.
- **Наблюдатель (контролёр)** — **до 5** на задачу, из списка зарегистрированных пользователей. Только просмотр. Видит **исключительно те задачи, где его назначили наблюдателем**, а не все задачи системы. Смысл — «поставить в копию» руководителя, чтобы он видел исполнение, не вмешиваясь («эффект наблюдателя»).

## 5. МОДЕЛЬ ЗАДАЧИ (поля)

- **Сквозной номер** — последовательная нумерация всех задач во всей системе, независимо от постановщика (1, 2, 3, …). Единый счётчик на всю систему.
- **Уникальный ID для поиска** — **6-значный** идентификатор, по которому можно искать задачу. Это **отдельное поле в дополнение** к сквозному номеру. *Проектное решение по реализации:* генерировать случайно в диапазоне `100000–999999` с проверкой уникальности (повторная генерация при коллизии), хранить с уникальным индексом.
- **Название** — текст, обязательное.
- **Описание** — текст.
- **Вложения** — **файл любого типа** («можно вложить что угодно») и/или **ссылка (URL)**. Вложений может быть **несколько** (несколько файлов и/или несколько ссылок).
- **Срок исполнения** — дата, с переключателем режима: **«дата + время»** либо **«только дата»**. Срок — обязательное поле (это следствие требований по напоминаниям и просрочке из раздела 9, а не отдельная оговорка директора).
- **Исполнитель (ответственный)** — один пользователь из реестра.
- **Постановщик** — автоматически (создатель задачи).
- **Наблюдатели** — 0–5 пользователей из реестра.
- **Статус задачи** — управляется **постановщиком**: `В работе` (стартовое значение при создании) / `Выполнена` / `Отменена`.
- **Флаг «Просрочена»** — **отдельное автоматическое булево поле**, НЕ статус. Выставляется системой автоматически по истечении срока, **не влияя** на статус: задача может быть одновременно `В работе` и `Просрочена`. Флаг — это **факт**: однажды возникнув, он не обнуляется автоматически и сохраняется в отчётности. После закрытия задачи (`Выполнена`/`Отменена`) прекращаются лишь **ежедневные напоминания** о просрочке (раздел 9), но сам факт просрочки не «снимается». Отображается как отдельная пометка (бейдж).
- **Информация о выполнении (отчёт исполнителя)** — поле «на стыке» исполнителя и постановщика: текст и/или вложение, которые добавляет **исполнитель**, отчитываясь о готовности. Отдельно от исходных полей задачи. Постановщик и наблюдатели её видят, но сами не пишут.
- **Отметка готовности от исполнителя** — флаг «исполнитель отметил готовность» (отписался о выполнении). Это НЕ статус `Выполнена` — это сигнал постановщику. Окончательный статус `Выполнена` ставит постановщик.
- Служебные поля: дата создания, дата изменения.

**Момент перехода в «Просрочена»** — *проектное решение* (директор точный момент не уточнял; согласуется с его словами «на следующий день у вас просроченная задача»):
- Режим «дата + время»: задача становится «Просрочена» по достижении указанного момента времени.
- Режим «только дата»: задача становится «Просрочена» с началом следующих суток после даты срока (т.е. в конце дня-срока она ещё не просрочена).
- Флаг должен быть достоверен независимо от частоты планировщика: вычисляй его «по факту» при чтении/отображении (`now ≥ момент_срока`) **или** лёгким частым проходом (см. раздел 9), а не только суточным прогоном уведомлений.

## 7. ЭКРАНЫ И НАВИГАЦИЯ (три ипостаси)

1. **Экран входа** — первый экран: e-mail + пароль; ссылки «Регистрация» и «Забыли пароль?»; предупреждение про список доступа.
2. **Главный экран** после входа — три раздела-переключателя (ипостаси) **постоянно доступны**, без отдельного промежуточного окна выбора роли, с переключением в любой момент. *Проектное решение по компоновке:* на десктопе — боковая панель **слева** (как просил директор); на узком/мобильном экране — адаптивно (верхние вкладки или бургер-меню).
   - **«Я постановщик»** → таблица всех задач, где пользователь постановщик, + кнопка **«Добавить задачу»**.
   - **«Я исполнитель»** → таблица всех задач, где пользователь исполнитель.
   - **«Я наблюдатель»** → таблица всех задач, где пользователь назначен наблюдателем (только свои).
3. На главном экране **желателен счётчик «дней до ближайшего дедлайна»** (по открытым задачам пользователя) — отражает цель директора «люди должны понимать, когда у них дедлайн».
4. **Клик по строке задачи** открывает карточку задачи с полями — для просмотра / заполнения / изменения в зависимости от роли и прав (см. матрицу прав).
5. **Поиск по 6-значному ID** — поле поиска, открывающее карточку найденной задачи. *Проектное решение:* область видимости поиска ограничивается правами роли (пользователь находит только задачи, к которым у него есть доступ как постановщику/исполнителю/наблюдателю) — по аналогии с приватностью наблюдателя; директор область поиска не уточнял.
6. Адаптивная вёрстка: корректно работает и читается с телефона.

## 8. СПИСКИ, СОРТИРОВКА, ВЫГРУЗКА

1. В каждом из трёх разделов — **таблица** с колонками по полям задачи: номер, ID, название, исполнитель/постановщик/наблюдатели (по контексту вкладки), срок, статус, флаг «Просрочена». Во вкладке **«Я постановщик»** добавь колонку-индикатор **«Готовность от исполнителя»** (отмечена/нет) — чтобы постановщик видел, кто уже отчитался, не открывая карточку.
2. **Сортировка по каждому полю** (клик по заголовку колонки).
3. **Сортировка по умолчанию — по дате** (ближайший срок выше). Это требование директора («автоматически сортировка должна выдаваться по дате»). *Проектное решение (предложение разработчика, директором явно не закреплено — уточнить):* в качестве вторичного ключа сначала показывать открытые/незакрытые (`В работе`) выше закрытых (`Выполнена`/`Отменена`), затем по дате.
4. Дополнительные сортировки: по дате, по исполнителю, по постановщику, по наблюдателю.
5. **Выгрузка** текущего списка (для любой из трёх ролей-вкладок) в файл, **отформатированный под печать**: аккуратная вёрстка, ничего не наезжает друг на друга, готово к печати и к пересылке кому-то (директор: «выгрузить, отправить кому-то») — НЕ «сырые машиночитаемые данные». *Проектное решение по формату:* директор требовал лишь «отформатировано под печать, чтобы не наезжало»; для self-host на скромном школьном сервере предпочтительно простое решение — печатная HTML-страница с корректным `print-CSS`; PDF — опционально. Выгружается текущий отфильтрованный/отсортированный список с читаемыми полями (номер, ID, название, описание/срок, исполнитель, постановщик, наблюдатели, статус, флаг просрочки, отчёт о выполнении).

## 9. УВЕДОМЛЕНИЯ (e-mail + MAX) И ГРАФИК НАПОМИНАНИЙ

**Каналы:**
- **E-mail — обязательный и гарантированный канал** и адресат по умолчанию. E-mail — обязательный реквизит каждой записи реестра.
- **MAX — где возможно** (best-effort). *Адресат MAX:* идентификатор пользователя в MAX (привязка: chat_id / телефон / бот) — **опциональное** поле в реестре/профиле, задаётся администратором или пользователем. Если привязки MAX нет — уведомление уходит **только по e-mail**.
- *Проектное решение по вложениям в MAX:* MAX принимает не все типы файлов (подтверждено: HTML не прошёл; «через почту тоже неплохо»). Для проблемных вложений отправляй ссылку и/или дублируй почтой.
- **Архитектура каналов — подключаемая (адаптеры/интерфейс канала),** чтобы позже без переделок добавить push и SMS. Сейчас реализуются только e-mail (обязательно) и MAX (best-effort при наличии API/бота).
- *Желаемый порядок для уведомления о постановке (директор):* «сначала на почту, потом в MAX» — почта первична и обязательна, MAX дополняет.

**График напоминаний (ровно 4 события):**
1. **В день постановки задачи** — уведомление получают **исполнитель И все наблюдатели**, с разным текстом: исполнителю — «Вам поставлена задача №… (вы назначены исполнителем)»; наблюдателю — «Вы назначены наблюдателем по задаче №…». С текстом задачи и вложениями.
2. **За сутки до срока** — уведомление **только исполнителю**: «Завтра истекает срок исполнения поручения №…».
3. **В день срока** — уведомление **только исполнителю**: «Срок вашей задачи №… истекает сегодня».
4. **После истечения срока** — задача помечается флагом **«Просрочена»**, и далее **каждый следующий день** исполнителю уходит напоминание, пока задача не закрыта (статус `Выполнена` или `Отменена`).

Наблюдатели получают только уведомление №1. Дальнейшие напоминания — только исполнителю.

**Правила планировщика (реализуемость):**
- **Раздели два процесса по частоте:**
  1. *Выставление флага «Просрочена»* — по факту (вычислять при чтении: `now ≥ момент_срока`) либо лёгким частым проходом (например, каждые 5–15 минут), чтобы режим «дата + время» срабатывал вовремя, а не ждал суточного прогона.
  2. *Рассылка плановых уведомлений* (события 1–4) — фоновый воркер, запускаемый **раз в сутки** в фиксированное время. *Проектное решение (с заказчиком не согласовано, вынести в конфиг):* время отправки по умолчанию, например `09:00` в таймзоне организации (`Europe/Moscow`).
- **Вычисление «за сутки» / «в день срока»:** для режима «только дата» — «за сутки» = прогон в день `(дата_срока − 1)`, «в день срока» = прогон в `дата_срока`; для режима «дата + время» — те же дни относительно даты момента.
- **Краевые случаи:** если задача создана в день срока (или позже) — уведомление о постановке (№1) всё равно уходит; уже неактуальное напоминание «за сутки» (№2) пропускается; в день срока уходит №3. За один прогон одной задаче — не более одного уведомления каждого типа в день.
- **Изменение срока постановщиком** (право дано в разделе 6): при смене срока задача переоценивается — флаг «Просрочена» снимается, если новый срок ещё не наступил; ключи идемпотентности уведомлений «за сутки»/«в день срока»/«ежедневно после срока» привязываются к версии срока (или сбрасываются при смене), чтобы напоминания сработали по новому сроку. Опционально — best-effort уведомление исполнителю об изменении срока.
- **Идемпотентность:** веди **журнал отправленных уведомлений** (по задаче + типу события + дате). Каждое плановое уведомление (1–3) отправляется ровно один раз за свой день; ежедневное напоминание о просрочке (4) — не более одного в день на задачу. Перезапуск воркера не порождает повторов.

## 10. ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ И ДЕПЛОЙ

- **Веб-приложение, адаптивное**, корректно работает с телефона. Мобильного приложения нет.
- **Self-hostable**, легко развернуть из репозитория **GitHub**. Любой может скачать и развернуть у себя бесплатно.

- **`.env.example`** со всеми переменными: SMTP (почта), параметры MAX, секреты/JWT, путь/креды к хранилищу вложений, таймзона, время отправки уведомлений, базовый URL, данные первичного администратора.

- **Стек (предложение, можно заменить с обоснованием):** backend — Python/FastAPI или Node/NestJS; БД — PostgreSQL; frontend — React + TypeScript. Для скромного школьного self-host предпочитай **простые решения**: встроенный cron-воркер (а не внешний брокер очередей) и **файловое хранилище** вложений (S3-совместимое — опционально, не обязательно).
- **Миграции БД** и **сид-скрипт** для первичного администратора и начального реестра почт.
- Загрузка вложений: тип не ограничивать («что угодно»), но задать разумный лимит размера и количества и выполнять базовую проверку (размер/имя — обязательно; антивирус-хук — опционально).
- **README** на русском: назначение, требования, быстрый старт через docker-compose, переменные окружения, развёртывание на школьном сервере (динамический IP), создание первичного администратора, настройка SMTP и MAX, лицензия (открытая/бесплатная).

### 10.1. Минимальный перечень REST-эндпойнтов (с проверкой роли на сервере)

Контракт API задан ниже как минимум; на этапе раздела 13 уточни детали. Для каждого эндпойнта обязательна серверная проверка прав.

- `POST /auth/register` — регистрация (проверка e-mail по реестру; вне реестра → фраза отказа).
- `POST /auth/login` — вход (e-mail + пароль).
- `POST /auth/password-reset/request` / `POST /auth/password-reset/confirm` — сброс пароля.
- `GET|POST|PUT|DELETE /admin/registry` — ведение реестра почт (**только админ**).
- `DELETE /admin/users/{id}` — удаление пользователя и всего его архива (**только админ**, требует подтверждения).
- `POST /tasks` — создать задачу (постановщик).
- `GET /tasks?role=author|assignee|observer` — списки задач по ипостаси текущего пользователя.
- `GET /tasks/{id}` — карточка задачи (с учётом прав видимости).
- `PUT /tasks/{id}` — изменить любое поле (только постановщик задачи).
- `DELETE /tasks/{id}` — удалить задачу (только постановщик).
- `PATCH /tasks/{id}/status` — сменить статус `В работе`/`Выполнена`/`Отменена` (только постановщик).
- `POST /tasks/{id}/report` — добавить информацию о выполнении: текст/вложение (только исполнитель).
- `POST /tasks/{id}/mark-ready` — отписаться о готовности (только исполнитель).
- `POST /tasks/{id}/attachments` — добавить вложение (постановщик — к задаче; исполнитель — к отчёту).
- `GET /tasks/search?code=NNNNNN` — поиск по 6-значному ID (в пределах прав видимости).
- `GET /tasks/export?role=...&format=print` — выгрузка текущего списка под печать.

## 11. КАЧЕСТВО И ТЕСТЫ

- **Серверная авторизация на каждом действии** — роль и права проверяются на бэкенде; ограничения нельзя обойти через прямые API-вызовы.
- **Журнал отправленных уведомлений — обязателен** (нужен для идемпотентности планировщика, раздел 9).
- **Полноценный аудит-лог всех действий** (создание/изменение/удаление, смена статуса, удаление пользователя и т.п.) — **опционально, не входит в ТЗ заказчика и не обязателен для MVP**. Делать только при наличии времени, не раздувая объём.
- **Автотесты на матрицу прав:**
  - исполнитель НЕ может редактировать поля задачи и менять статус;
  - наблюдатель — только чтение, видит только назначенные ему задачи;
  - постановщик — полный доступ к своим задачам; отчёт о выполнении добавляет только исполнитель.
- **Автотесты на бизнес-логику:**
  - сквозная нумерация (последовательность, отсутствие дыр/дублей) и уникальность 6-значного ID (включая повтор генерации при коллизии);
  - флаг «Просрочена» выставляется автоматически и не меняет статус (задача `В работе` + `Просрочена`);
  - корректность срабатывания просрочки в режимах «дата+время» и «только дата», в т.ч. при изменении срока;
  - график из 4 уведомлений: правильные получатели и тайминги, краевые случаи, идемпотентность (нет дублей при повторном запуске планировщика);
  - удаление пользователя из реестра удаляет его архив и блокирует вход;
  - отказ в регистрации для e-mail вне реестра показывает ровно «Извините, у вас нет доступа к сервису.».
- Валидация на сервере: ровно один исполнитель, не более 5 наблюдателей, обязательные название и срок.

- **Подключаемые каналы уведомлений** (адаптеры) с обязательным e-mail и best-effort MAX — задел под push/SMS без переписывания (раздел 9).

Поскольку подразделы описывают разные слои, один и тот же объект называется по-разному на уровне БД, API и UI. Каноническое соответствие (источник истины для реализации):

Иллюстративные SQL/псевдокод-фрагменты могут использовать русские метки статуса (`'В работе'`) для читаемости — каноническими остаются enum-коды.

#### 13.1.1. Обоснование выбора стека

Ключевой контекст из ТЗ: **одна организация, узкий круг пользователей** (штат школы — десятки, максимум первые сотни человек), **скромный школьный self-host** (раздел 10), развёртывание «одной командой» энтузиастом, а не DevOps-командой, и срок-ориентир — демонстрация в середине июня. Отсюда главный критерий выбора — **минимум движущихся частей и минимум эксплуатационной нагрузки**, а не масштабируемость на тысячи RPS.

| Слой | Выбор | Почему именно он | Что сознательно НЕ берём |
|---|---|---|---|
| Backend | **Python 3.12 + FastAPI** | Декларативная валидация на Pydantic закрывает требование «серверная проверка на каждом действии» (раздел 11) почти бесплатно: схемы запросов/ответов = контракт + валидатор (ровно один исполнитель, ≤5 наблюдателей, обязательные поля). Автогенерация OpenAPI/Swagger UI — готовая документация API для других школ. ASGI достаточно производителен для нашей нагрузки. Низкий порог входа для сопровождающего из школы. | Django (тяжёлый ORM+admin, избыточен под плоскую модель и кастомные роли-на-задачу); Node/NestJS (допустим по ТЗ, но Python проще читается непрофильным сопровождающим и даёт более лаконичный cron-воркер) |
| БД | **PostgreSQL 16** | Нужны: транзакционно-надёжный **сквозной счётчик** задач (раздел 5), **уникальный индекс** на 6-значном ID, частичные индексы под фильтры вкладок, **журнал уведомлений** с уникальным ключом идемпотентности (раздел 9). Всё это — штатный реляционный функционал. JSON-поля Postgres дают «архитектурный задел» (push/SMS, будущий «проект») без смены СУБД. | SQLite (нет надёжных конкурентных upsert'ов для журнала идемпотентности, риск блокировок при воркере+API одновременно); любые NoSQL (данные строго реляционные) |

Адаптивная вёрстка под десктоп (боковая панель слева) и телефон (раздел 7) — стандартная задача для React. Сборка Vite → статика, отдаётся nginx (см. 13.1.3). | SSR-фреймворки (Next.js) — не нужны: это внутренний инструмент за авторизацией, SEO/SSR не требуется, лишний Node-рантайм в проде |
| Планировщик | **APScheduler внутри отдельного контейнера worker** (тот же образ, что backend) | Встроенный cron-воркер по ТЗ (раздел 10) — без внешнего брокера. См. обоснование «один образ / отдельный контейнер» в 13.1.4. | Celery+Redis/RabbitMQ (внешний брокер прямо назван избыточным в ТЗ); системный cron хоста (ломает «docker compose up», не переносим) |
| Хранилище вложений | **Файловый том (volume)**, S3-совместимость — опционально через адаптер | ТЗ: файловое хранилище предпочтительно (раздел 10). «Вложить что угодно» (раздел 5) → храним как непрозрачные файлы на диске, метаданные — в БД. | Обязательный MinIO/S3 (лишний контейнер и сложность для школьного сервера; оставляем как опциональный адаптер) |
| Reverse-proxy / TLS | **Caddy** (по умолчанию) | Автоматический HTTPS из коробки (Let's Encrypt) одной строкой конфига — критично, так как сопровождающий не DevOps. > **Проектное решение (reverse-proxy).** ТЗ перечисляет контейнеры, но не фиксирует конкретный прокси. Предлагается **Caddy** ради автоматического TLS под школьный self-host с доменом. Если у школы уже есть свой nginx/туннель — Caddy-сервис из `docker-compose.yml` можно отключить (профиль `--profile noproxy`), backend и фронт от этого не зависят. **Требует подтверждения заказчиком.**

**Почему НЕ «тяжёлые» решения в целом.** Микросервисы, Kubernetes, отдельный брокер очередей, объектное хранилище и SSR — это операционные издержки (мониторинг, обновления, отладка), которые на школьном сервере под десятки пользователей не окупаются и противоречат требованию «развернуть одной командой». Архитектура — **модульный монолит** (backend как один FastAPI-процесс с разделением на слои/модули) + тонкий воркер. Это даёт простоту эксплуатации при сохранении чистых границ для будущих расширений.

#### 13.1.2. Диаграмма компонентов и контейнеров

Все контейнеры — в одной Docker-сети `internal`. Наружу (в `external`) смотрит только reverse-proxy. БД, backend и worker **не публикуют порты на хост** — доступ к ним только внутри сети.

```
                       Интернет / ЛВС школы
                               │  HTTPS (443) / HTTP→HTTPS редирект (80)
                               ▼
        ┌───────────────────────────────────────────────────┐
        │            reverse-proxy  (Caddy)                  │
        │   TLS-терминация, единая точка входа               │
        │     /         → frontend  (статика)                │
        │     /api/*    → backend   (REST, HTTP)             │
        └───────────────┬───────────────────┬───────────────┘
                        │ HTTP              │ HTTP /api
                        ▼                   ▼
            ┌────────────────────┐  ┌────────────────────────────┐
            │  frontend          │  │  backend (API)             │
            │  nginx + статика    │  │  FastAPI / uvicorn         │
            │  React+TS (build)  │  │  REST, авторизация по JWT  │
            └────────────────────┘  └──────┬───────────┬─────────┘
                                            │ SQL       │ файлы (R/W)
                                            │ :5432     │
                                            ▼           ▼
                              ┌──────────────────┐  ┌──────────────────────┐
                              │  db (PostgreSQL) │  │  volume: attachments │
                              │  том pgdata      │  │  (файлы вложений)     │
                              └──────┬───────────┘  └──────────┬───────────┘
                                     ▲ SQL :5432               ▲ файлы (R/W)
                                     │                         │
                              ┌──────┴─────────────────────────┴───────────┐
                              │  worker / scheduler                        │
                              │  тот же образ, что backend; APScheduler:   │
                              │   • частый проход — флаг «Просрочена»       │
                              │   • суточная рассылка уведомлений (4 события)│
                              │  → исходящий SMTP (e-mail) / MAX API        │
                              └────────────────────────┬───────────────────┘
                                                       │ SMTP / HTTPS
                                                       ▼
                                          внешний SMTP-сервер,  MAX API/бот
```

**Кто с кем общается и по каким протоколам:**

| Источник | Назначение | Протокол | Назначение связи |
|---|---|---|---|
| Браузер | reverse-proxy | HTTPS | Единственная публичная точка входа |
| reverse-proxy | frontend | HTTP | Отдача собранной статики React |
| reverse-proxy | backend | HTTP | Проксирование `/api/*` |
| backend | db | PostgreSQL wire (TCP 5432) | Чтение/запись данных |
| backend | volume `attachments` | файловая ФС | Сохранение/отдача вложений |
| worker | db | PostgreSQL wire (TCP 5432) | Чтение задач, запись флага просрочки, запись журнала уведомлений |
| worker | volume `attachments` | файловая ФС | Чтение файлов для вложения в письма (см. подраздел уведомлений по обработке вложений для MAX) |
| worker | внешний SMTP | SMTP/SMTPS | E-mail-уведомления (обязательный канал) |
| worker | MAX API/бот | HTTPS | MAX-уведомления (best-effort) |

> **Примечание о видимости вложений (реверс-прокси).** Файлы из тома `attachments` отдаются **только через backend** (эндпойнт скачивания с серверной проверкой прав видимости — см. матрицу прав, раздел 6, и подраздел API), а не статикой напрямую. Это исключает обход авторизации по прямой ссылке на файл.

#### 13.1.3. Контейнер frontend

`frontend` собирается двухстадийно: стадия `node` (Vite build) → стадия `nginx`, отдающая статику. Внутренний nginx фронта **не** занимается TLS и внешней маршрутизацией — он лишь раздаёт SPA и отдаёт `index.html` на неизвестные пути (SPA-fallback для клиентского роутинга). Внешний reverse-proxy (Caddy) стоит перед ним. Базовый URL API фронт берёт из переменной сборки/рантайма (см. `BASE_URL` / `API_BASE_URL` в 13.1.6).

#### 13.1.4. Один процесс vs несколько: где живёт планировщик

**Решение: backend и worker — это два контейнера из ОДНОГО Docker-образа, но с разными командами запуска (entrypoint).**

```
один образ  ──► backend-сервис:  uvicorn app.main:app   (обслуживает HTTP)
            └─► worker-сервис:   python -m app.worker   (APScheduler, без HTTP)
```

Обоснование:

- **Один образ — единая кодовая база и зависимости.** Воркер использует те же модели, репозитории и доменные сервисы, что и API (вычисление момента просрочки, формирование текста уведомлений, журнал идемпотентности). Дублировать образ/код нельзя — рассинхрон логики просрочки между API и воркером недопустим (флаг должен быть достоверен и «по факту при чтении», и при фоновом проходе — раздел 5, 9). Один Dockerfile → собирается один раз, оба сервиса ссылаются на него.
- **Два контейнера, а не один процесс, — для изоляции отказов и нагрузки.** Если планировщик завис на медленной SMTP-рассылке или MAX-таймауте, это **не должно** задерживать HTTP-ответы пользователям, и наоборот. Отдельный контейнер можно перезапускать (`restart: unless-stopped`) и масштабировать API независимо. Падение воркера не роняет API.
- **Ровно один экземпляр воркера.** Планировщик запускается в **единственном** реплике (в `docker-compose` сервис `worker` не масштабируется). Это вместе с **журналом идемпотентности в БД** (раздел 9: ключ `задача + тип события + дата`, уникальный индекс) гарантирует, что повторный запуск/перезапуск воркера не порождает дублей уведомлений. Даже если кто-то по ошибке поднимет два воркера, уникальный индекс журнала не даст отправить уведомление дважды (вставка-маркер до отправки).

> **Проектное решение (внутренняя структура воркера).** APScheduler в одном процессе worker ведёт **две независимые джобы**: (а) *частый проход* (интервал из конфига, по умолчанию каждые 10 минут) — пересчёт флага «Просрочена» для режима «дата+время», см. подраздел про логику просрочки; (б) *суточная джоба* в фиксированное время (`NOTIFY_TIME`, по умолчанию `09:00 Europe/Moscow`) — рассылка 4 плановых событий. Время и интервал вынесены в `.env`. Детальный алгоритм каждой джобы — в смежных подразделах (просрочка / уведомления). **Время суточной рассылки и таймзона требуют подтверждения заказчиком** (отмечено в разделе 9 ТЗ).

#### 13.1.6. Управление конфигурацией через `.env`

Вся конфигурация — через переменные окружения (12-factor), шаблон `.env.example` коммитится, реальный `.env` — нет. Backend читает их через `pydantic-settings` (`config.py`), что даёт **валидацию конфига на старте** (упавший контейнер при отсутствии обязательной переменной — лучше, чем тихая ошибка). Ключевые группы:

| Группа | Переменные (пример) | Назначение |
|---|---|---|
| **БД** | `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`, `DATABASE_URL` | Подключение backend/worker к Postgres; креды тома `pgdata` |
| **JWT / секреты** | `JWT_SECRET`, `JWT_ACCESS_TTL`, `JWT_REFRESH_TTL`, `PASSWORD_RESET_TTL` | Подпись токенов сессии и **токена сброса пароля со сроком жизни** (раздел 3) |
| **SMTP (e-mail, обязательный канал)** | `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_TLS`, `SMTP_FROM` | Гарантированный канал уведомлений (раздел 9) |
| **MAX (best-effort)** | `MAX_ENABLED`, `MAX_API_BASE`, `MAX_BOT_TOKEN` | Опциональный канал; при `MAX_ENABLED=false` или отсутствии привязки — только e-mail |
| **Хранилище вложений** | `STORAGE_BACKEND` (`file`\|`s3`), `ATTACHMENTS_DIR`, (опц.) `S3_ENDPOINT`/`S3_BUCKET`/`S3_KEY`/`S3_SECRET` | Путь к тому или S3-опция (адаптер `storage/`) |
| **Лимиты загрузки (MAX-*)** | `MAX_UPLOAD_SIZE_MB`, `MAX_ATTACHMENTS_PER_TASK`, `MAX_OBSERVERS` (=5) | Разумные лимиты размера/количества вложений и наблюдателей (раздел 10, 11) |
| **Таймзона и расписание** | `TZ` (=`Europe/Moscow`), `NOTIFY_TIME` (=`09:00`), `OVERDUE_SCAN_INTERVAL_MIN` (=`10`) | Таймзона организации; время суточной рассылки; интервал частого прохода просрочки (раздел 9) |
| **Адрес/доступ** | `BASE_URL`, `CORS_ORIGINS` | Базовый URL для ссылок в письмах (сброс пароля, ссылки на задачи) и работы за динамическим IP/доменом (раздел 10) |
| **Первичный администратор** | `ADMIN_EMAIL`, `ADMIN_PASSWORD` (или `ADMIN_INITIAL_TOKEN`) | Сид-скрипт создаёт первичного админа при развёртывании (раздел 3, 10) |
| **Окружение** | `APP_ENV` (`dev`\|`prod`), `LOG_LEVEL` | Режим работы, уровень логирования |

> **Проектное решение (первичный администратор).** Безопаснее задавать `ADMIN_EMAIL` + одноразовый `ADMIN_INITIAL_TOKEN` и завершать создание пароля через обычную регистрацию/сброс (пароль не лежит в `.env` в открытом виде). Но ТЗ (раздел 3, 10) допускает прямое `ADMIN_PASSWORD`. По умолчанию поддержим оба пути, в `README` рекомендуем токен-вариант для прода. **Согласовать с заказчиком.**

**Связь с docker-compose.** `docker-compose.yml` пробрасывает эти переменные в сервисы: `db` получает группу БД; `backend` и `worker` — все группы (общий образ); `reverse-proxy` — домен/`BASE_URL` для TLS; `frontend` — `API_BASE_URL`/`BASE_URL` на стадии сборки/рантайма. Том `attachments` монтируется в `backend` и `worker` по пути `ATTACHMENTS_DIR`; том `pgdata` — в `db`.

### 13.2. Модель данных и ERD

Подраздел описывает реляционную модель PostgreSQL для системы поручений школы № 2090. Проектируется строго в рамках раздела 2 ТЗ: задачи **плоские**, исполнитель **ровно один**, наблюдателей **≤ 5**, «Просрочена» — **булев флаг**, а не статус, роли назначаются **на задачу**, а не на аккаунт (поэтому в схеме нет глобальной таблицы ролей — роль выводится из позиции пользователя в конкретной задаче).

Вопросы аутентификации/потоков сброса пароля относятся к смежному подразделу «Аутентификация и авторизация»; здесь описываются только хранимые сущности `users`, `password_reset_tokens`, `email_registry` в части их **полей и инвариантов**. Логика планировщика и каналов — смежный подраздел «Уведомления»; здесь описываются только таблицы `notification_log` и каналы как enum/поля.

**Наблюдатели (M:N, ≤ 5).**

> **Проектное решение (лимит ≤ 5).** Жёсткого «ровно 5» в БД нет — это динамический агрегат. Лимит проверяется (а) в приложении при добавлении наблюдателя и (б) подстраховкой на БД через триггер-счётчик, чтобы лимит нельзя было обойти прямым INSERT в обход API (раздел 6 «нельзя обойти через API»):

```sql
CREATE FUNCTION trg_observers_limit() RETURNS trigger AS $$
BEGIN
  IF (SELECT count(*) FROM task_observers WHERE task_id = NEW.task_id) > 5 THEN
     RAISE EXCEPTION 'OBSERVERS_LIMIT_EXCEEDED: max 5 observers per task';
  END IF;
  RETURN NEW;
END $$ LANGUAGE plpgsql;

CREATE CONSTRAINT TRIGGER observers_limit
  AFTER INSERT ON task_observers
  DEFERRABLE INITIALLY IMMEDIATE
  FOR EACH ROW EXECUTE FUNCTION trg_observers_limit();
```

#### 13.2.5. Стратегия сквозного номера (`task_no`)

Требование: единый монотонный счётчик на всю систему **без дыр и дублей** (раздел 5, тест 11).

> **Проектное решение.** Используем `CREATE SEQUENCE seq_task_no` + `task_no UNIQUE`. Однако стандартный `SEQUENCE` гарантирует только **отсутствие дублей и монотонность**, но не отсутствие дыр (откат транзакции «теряет» выданное значение). ТЗ требует «отсутствие дыр». Поэтому сквозной номер присваивается **в момент успешной фиксации создания задачи** через таблицу-счётчик с блокировкой строки, а не через `nextval`:

```sql
CREATE TABLE counters (
    name        TEXT PRIMARY KEY,
    value       BIGINT NOT NULL DEFAULT 0
);
INSERT INTO counters(name, value) VALUES ('task_no', 0);
```

Внутри транзакции создания задачи:

```sql
-- сериализуем выдачу номера: блокировка строки счётчика
UPDATE counters SET value = value + 1 WHERE name='task_no' RETURNING value;  -- → next_no
INSERT INTO tasks(task_no, code6, ...) VALUES (next_no, ...);
COMMIT;  -- откат вернёт и счётчик, и INSERT → дыры не возникает
```

Сравнение подходов:

| Подход | Дубли | Дыры | Конкурентность |
|---|---|---|---|
| `SEQUENCE` (`nextval`) | нет | возможны при откате | высокая (без блокировок) |
| **Таблица-счётчик + `UPDATE…RETURNING`** | нет | **нет** (откат возвращает значение) | сериализация по одной строке — приемлемо для школьного объёма |

Для скромной нагрузки школы (десятки задач в день) сериализация выдачи номера незаметна. `task_no UNIQUE` остаётся страховкой целостности. В DDL выше `DEFAULT nextval(...)` показан как запасной вариант; **рабочая стратегия — таблица-счётчик**.

#### 13.2.6. Стратегия 6-значного ID (`code6`)

Требование: случайный код `100000–999999`, проверка уникальности, повтор при коллизии, уникальный индекс — **отдельное** поле помимо `task_no` (раздел 5).

```python
def generate_code6(session) -> int:
    for _ in range(10):                       # ретраи на случай коллизий
        code = secrets.randbelow(900_000) + 100_000   # 100000..999999
        if not session.query(Task.id).filter(Task.code6 == code).first():
            return code
    raise ConflictError("CODE6_EXHAUSTED")    # практически недостижимо при 900k диапазоне
```

Гонку закрывает БД: при параллельной вставке одинакового `code6` сработает `UNIQUE`, приложение ловит `IntegrityError` и повторяет генерацию (insert-retry-loop). Пространство 900 000 значений против сотен задач школы → коллизии единичны.

> **Проектное решение.** `task_no` и `code6` — **разные поля разного назначения**: `task_no` — человекочитаемый порядковый («задача №7»), `code6` — непоследовательный поисковый код (не раскрывает количество задач, удобен для устной передачи). Не объединять.

#### 13.2.7. Инварианты (БД + приложение)

| Инвариант | Уровень БД | Уровень приложения |
|---|---|---|
| Ровно один исполнитель | `assignee_id NOT NULL` (скаляр, не M:N) | проверка существования и «не удалён» |
| ≤ 5 наблюдателей | constraint-триггер `observers_limit` | проверка перед добавлением + дедуп по PK |
| Обязательное «название» | `CHECK length(btrim(title))>0` | валидация формы |
| Обязательный «срок» | `due_at NOT NULL` | валидация формы |
| Режим срока | `due_mode NOT NULL` + нормализация `due_at` | переключатель «дата+время/только дата» |
| `created_at`/`updated_at` | `DEFAULT now()` | `updated_at` через ORM-хук/триггер при UPDATE |
| Уникальность `task_no`/`code6` | `UNIQUE` | счётчик / insert-retry |
| Флаг `is_overdue` | `bool NOT NULL DEFAULT false` | выставляется планировщиком + «по факту» при чтении (`now ≥ due_at`); `overdue_since` не сбрасывается (факт) |
| Статус ≠ просрочка | разные колонки `status`/`is_overdue` | задача может быть `in_progress` + `is_overdue=true` |
| Исполнитель ≠ удалённый пользователь | FK `RESTRICT` | при удалении — обезличивание (см. 13.2.9) |

Триггер `updated_at`:

```sql
CREATE FUNCTION set_updated_at() RETURNS trigger AS $$
BEGIN NEW.updated_at = now(); RETURN NEW; END $$ LANGUAGE plpgsql;
CREATE TRIGGER tasks_touch BEFORE UPDATE ON tasks
  FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

#### 13.2.9. Удаление пользователя из реестра: каскад своего архива + обезличивание чужих следов

Требование (раздел 3, п.10–11): удаление пользователя из реестра удаляет **весь его архив** (задачи, где он постановщик, и его вложения/данные) и при этом **не должно ломать целостность чужих задач**, где он был исполнителем/наблюдателем. ТЗ помечает объём «архива» как **проектное решение на согласование**. Предлагается следующий механизм, реализующий вариант «по умолчанию» из ТЗ (пометить как «пользователь удалён»):

**Шаги операции `DELETE /admin/users/{id}` (одна транзакция):**

1. **Каскад собственного архива (hard-delete).** Все `tasks` с `author_id = X` удаляются физически; за ними каскадно (`ON DELETE CASCADE`) уходят их `task_observers`, `task_attachments`, `task_reports`, `report_attachments`, `notification_log`. Файлы вложений этих задач удаляются из файлового хранилища отдельным шагом (по списку `file_path`, собранному до удаления строк).
2. **Обезличивание в чужих задачах, где X — исполнитель (tombstone).** FK `assignee_id` объявлен `ON DELETE RESTRICT`, поэтому строку `users` нельзя удалить, пока на неё ссылаются. Вместо физического удаления `users` помечаем запись как tombstone:
   ```sql
   UPDATE users
      SET is_deleted = true,
          deleted_at = now(),
          display_name = 'Пользователь удалён',
          email = 'deleted+' || id || '@removed.local',  -- освобождаем e-mail для повторного допуска
          password_hash = ''                              -- вход невозможен
    WHERE id = X;
   ```
   Чужие задачи сохраняются; в карточке исполнитель отображается как «Пользователь удалён» (через `users.display_name`). Это и есть **soft-reference / tombstone**: ссылка цела, личность обезличена.
3. **Обезличивание в чужих задачах, где X — наблюдатель.** Строки `task_observers` для X **не удаляем** (иначе «исчезнет» факт участия), а полагаемся на снапшот `task_observers.display_name`, который при удалении перезаписываем:
   ```sql
   UPDATE task_observers SET display_name = 'Пользователь удалён' WHERE user_id = X;
   ```
4. **Реестр.** Запись в `email_registry` удаляется (освобождая e-mail), что и блокирует повторный вход (раздел 3, п.11): серверная проверка доступа на каждый запрос требует наличие e-mail в реестре И `users.is_deleted = false`.

> **Проектное решение (вынести на подтверждение заказчику — раздел 3, п.51).** «Снять / обезличить / пометить» след в чужих задачах строго не задан. Реализовано **обезличивание (tombstone)** как наименее разрушающее целостность: чужие задачи и их история отчётности сохраняются, а персональные данные удаляемого исчезают. Альтернативы на согласование: (а) полностью удалять X из чужих задач как наблюдателя; (б) при удалении X-исполнителя автоматически переводить чужие задачи в `Отменена`. По умолчанию — вариант, описанный выше.

**Почему snapshot `display_name` в `task_observers` и `tasks` (через `users.display_name`).** После tombstone строка `users` остаётся, поэтому имя исполнителя читается из неё. Для наблюдателей снапшот хранится прямо в связи, чтобы не делать join к возможно изменённому `users` и сохранить отображение «Пользователь удалён» независимо.

#### 13.2.10. Ключевые индексы (списки и сортировки)

Под три вкладки (раздел 7) и сортировку по умолчанию «ближайший срок выше», с рекомендованным вторичным ключом «открытые выше закрытых» (раздел 8):

```sql
-- Вкладка «Я постановщик»: список + сортировка по сроку
CREATE INDEX ix_tasks_author_due   ON tasks(author_id, due_at);
-- Вкладка «Я исполнитель»
CREATE INDEX ix_tasks_assignee_due ON tasks(assignee_id, due_at);
-- Вкладка «Я наблюдатель» — через ix_observers_user(user_id) + join по task_id
-- Поиск по 6-значному коду (в пределах прав видимости)
-- покрыт UNIQUE(code6); UNIQUE(task_no) покрывает поиск/сортировку по номеру

-- Сортировка «статус → срок» (рекомендация раздела 8): частичный индекс открытых задач
CREATE INDEX ix_tasks_open_due ON tasks(due_at)
  WHERE status = 'in_progress';

-- Планировщик: быстрый отбор кандидатов на просрочку и напоминания
CREATE INDEX ix_tasks_due_open ON tasks(due_at)
  WHERE status = 'in_progress';
CREATE INDEX ix_tasks_overdue  ON tasks(is_overdue)
  WHERE is_overdue = true AND status = 'in_progress';
```

> **Проектное решение.** «Сортировка открытые→закрытые, затем по дате» реализуется в запросе как `ORDER BY (status='in_progress') DESC, due_at ASC`; частичный индекс `ix_tasks_open_due` ускоряет основной кейс (просмотр открытых). Для объёмов школы этого достаточно; составные индексы по `author_id/assignee_id` покрывают фильтрацию по вкладкам.

#### 13.3.1. Слоистая структура (routers → services → domain → repositories)

Принят классический четырёхслойный расклад. Цель — чтобы **бизнес-правила и инварианты жили в одном месте** и не могли быть обойдены ни через альтернативный эндпойнт, ни через прямой вызов БД. Каждый слой зависит только от слоёв ниже; зависимость вниз идёт через DI FastAPI (`Depends`).

```
HTTP ─▶ routers/        тонкий слой: парсинг, Pydantic-схемы (DTO), HTTP-коды,
        (api layer)     подключение dependency-guards. БИЗНЕС-ЛОГИКИ НЕТ.
          │
          ▼
        services/       сценарии/use-cases: оркестрация, транзакции (unit of work),
        (application)   вызов domain-правил, обращение к repositories,
                        постановка задач в очередь уведомлений (outbox).
          │
          ▼
        domain/         чистые правила и инварианты: «исполнитель ровно один»,
        (core)          «наблюдателей ≤ 5», вычисление роли пользователя по задаче,
                        матрица «роль → действие», статусные переходы, момент
                        просрочки. БЕЗ I/O, БЕЗ FastAPI, БЕЗ SQLAlchemy.
          │
          ▼
        repositories/   доступ к БД (SQLAlchemy 2.x): CRUD, выборки списков,
        (infrastructure)атомарное взятие sequence-номера, генерация уникального
                        6-значного кода. SQL-детали не «протекают» вверх.
```

**Распределение ответственности по слоям:**

| Слой | Отвечает за | НЕ делает |
|---|---|---|
| `routers/` | маршруты, валидация формы запроса (Pydantic), сериализация ответа, подключение guard-зависимостей, маппинг доменных ошибок → HTTP | не содержит правил доступа и бизнес-логики, не ходит в БД напрямую |
| `services/` | сценарий целиком, границы транзакции, запись в `notification_outbox`, аудит (если включён), вызов domain для проверок | не знает про HTTP-объекты (`Request`/`Response`), не формирует HTTP-коды |
| `domain/` | инварианты, статус-машина, расчёт `role_of(user, task)`, политика прав `can(action, role)`, момент перехода в «Просрочена» | не делает запросов и сетевых вызовов; на вход получает уже загруженные данные |
| `repositories/` | запросы/команды к PostgreSQL, конкурентно-безопасная нумерация, индексы уникальности | не принимает решений о правах и валидности |

**Почему именно так (обоснование).** ТЗ (разделы 6 и 11) требует, чтобы **ни одно ограничение нельзя было обойти через API**. Если правила разбросать по роутерам, появление нового эндпойнта (например, будущий `PATCH` для частичного редактирования) рискует «забыть» проверку. Вынося их в `domain/` и заставляя `services/` вызывать их **единообразно**, мы получаем единственный путь принятия решения о доступе и валидности. Слой `domain` чист (без I/O) — его покрывают быстрые unit-тесты матрицы прав (раздел 11), не поднимая БД.

**Пример прохода запроса** (`PATCH /tasks/{id}/status`):

```
router.update_status(task_id, body, ctx=Depends(get_task_context))
  └─ service.change_status(ctx, new_status)
       ├─ guard уже выполнен в dependency (см. 13.3.4): ctx.role == AUTHOR
       ├─ domain.status_machine.validate(task.status → new_status)
       ├─ repo.tasks.update_status(...)            # в одной транзакции
       └─ outbox.enqueue(...)  # если переход требует уведомления (см. 13.4)
```

**Карта каталогов** (ориентир, не жёсткое требование):

```
app/
  api/            # routers + Pydantic-схемы запросов/ответов
    deps/         # dependency-guards: get_current_user, get_task_context, require_admin
  services/       # auth_service, task_service, registry_service, report_service
  domain/         # roles.py, permissions.py, invariants.py, status.py, overdue.py
  repositories/   # users_repo, registry_repo, tasks_repo, tokens_repo, outbox_repo
  db/             # модели SQLAlchemy, сессия, миграции Alembic
  core/           # конфиг (pydantic-settings), безопасность (хэш/JWT), время/таймзона
```

#### 13.3.2. Аутентификация: выбор схемы для self-host

> **Проектное решение (рекомендация разработчика, согласовать).** Выбор между JWT и серверными сессиями ТЗ оставляет открытым. Ниже — обоснованный выбор в пользу **JWT (access + refresh) с refresh-токеном в httpOnly-cookie и хранением допустимых refresh-токенов в БД**. Это даёт мгновенный отзыв доступа (требование 3.11–3.12) при простоте развёртывания.

**Сравнение для нашего контекста (одна школа, скромный self-host, без внешних брокеров):**

| Критерий | Чистый stateless JWT | Серверные сессии (БД/Redis) | **Принятый гибрид: JWT + refresh в БД** |
|---|---|---|---|
| Доп. инфраструктура | нет | нужен стор сессий (Redis/таблица) | таблица в **той же** PostgreSQL — нового сервиса нет |
| Мгновенный отзыв (удаление из реестра) | ❌ access живёт до истечения | ✅ | ✅ короткий access + проверка реестра на каждый запрос (13.3.3) |
| Масштаб горизонтально | ✅ | зависит от стора | ✅ (для одной школы не критично) |
| Сложность | низкая | средняя | низкая-средняя |

**Итоговая схема токенов:**

| Токен | Время жизни (конфиг) | Где хранится | Назначение |
|---|---|---|---|
| **access** (JWT) | короткое, по умолч. **15 мин** | в памяти SPA (не в localStorage) | авторизация запросов, `Authorization: Bearer` |
| **refresh** (непрозрачный, случайный) | по умолч. **14 дней** | **httpOnly + Secure + SameSite=Strict** cookie | обновление access через `POST /auth/refresh` |

**Почему refresh — в httpOnly-cookie, а access — в памяти.** Защита от XSS: JS не имеет доступа к refresh-токену. Access короткоживущий и хранится только в памяти вкладки (теряется при перезагрузке — восстанавливается через `refresh`). Это стандартная практика и не требует от школьного админа ничего, кроме HTTPS (см. **13.7 «Тестирование, миграции, сид и развёртывание»** о reverse-proxy/TLS).

**Хранение refresh в БД** (таблица `refresh_tokens`): храним **хэш** токена, `user_id`, `expires_at`, `revoked_at`. Это даёт:
- немедленный logout / «выйти на всех устройствах» (пометить `revoked_at`);
- каскадный отзыв при удалении пользователя из реестра (см. 13.3.3);
- ротацию: при каждом `refresh` старый токен инвалидируется, выдаётся новый (защита от повторного использования).

**Состав JWT (claims):** `sub` (user_id), `email`, `is_admin` (срез на момент выдачи — но решения по админ-действиям всё равно перепроверяются по БД, см. 13.3.5), `iat`, `exp`, `jti`. Подпись — **HS256**, секрет из `JWT_SECRET` (`.env`). Для одной инсталляции асимметрия (RS256) избыточна.

**Хэширование паролей** — `argon2id` (через `passlib`/`argon2-cffi`), параметры в конфиге. Альтернатива — `bcrypt`; argon2id предпочтительнее как современный дефолт. Пароли в открытом виде/обратимом виде не хранятся никогда.

#### 13.3.3. Проверка членства e-mail в реестре на КАЖДЫЙ запрос (мгновенная блокировка)

Это центральное требование разделов 3.11–3.12: пользователь, удалённый из реестра, **теряет доступ немедленно**, а не после истечения токена. Реализуется **единой FastAPI-зависимостью** `get_current_user`, подключённой ко всем защищённым роутерам.

```python
async def get_current_user(
    creds: HTTPAuthorizationCredentials = Depends(bearer_scheme),
    db: AsyncSession = Depends(get_db),
) -> CurrentUser:
    payload = decode_access_jwt(creds.credentials)        # 401 если подпись/exp невалидны
    user = await users_repo.get_active_by_id(db, payload.sub)
    # КЛЮЧЕВАЯ ПРОВЕРКА: членство в реестре + связка с активной учёткой
    if user is None or not await registry_repo.is_listed(db, user.email):
        raise HTTPException(401, detail="ACCESS_REVOKED")  # токен бесполезен
    return CurrentUser(id=user.id, email=user.email, is_admin=user.is_admin)
```

**Связка модели** (детали в 13.2): запись реестра `registry_entry(email UNIQUE, max_id?, ...)` и учётная запись `user(email UNIQUE → FK на реестр, password_hash, is_admin, ...)`. Проверка `is_listed` смотрит **наличие записи в реестре** по e-mail. Так удаление строки реестра автоматически закрывает вход и доступ.

**Что происходит при удалении пользователя из реестра** (`DELETE /admin/users/{id}`, раздел 3.10) — атомарно в одной транзакции `registry_service`:
1. отзыв всех `refresh_tokens` пользователя (`revoked_at = now`) → новый access получить нельзя;
2. удаление записи реестра/учётки → текущий короткоживущий access **перестаёт проходить `get_current_user`** на следующем же запросе (макс. задержка = TTL access, по умолч. 15 мин; refresh уже мёртв);
3. удаление «архива» — см. ниже проектное решение.

> **Проектное решение (раздел 3.10, требует подтверждения заказчиком).** Объём «архива» при удалении пользователя не уточнён. Предлагается: **удалять задачи, где он постановщик** (`author_id`), со всеми их вложениями/отчётами; в **чужих** задачах, где он был исполнителем или наблюдателем, — **не удалять задачу**, а обезличивать участие (исполнитель → специальный маркер «пользователь удалён», наблюдатель → исключить из списка), чтобы не рушить целостность чужих задач. На согласование: снять / обезличить / пометить. Техническая основа: FK на `author_id` — `ON DELETE CASCADE` (через сервисный слой, контролируемо), FK на `assignee_id`/`observers` — `SET NULL`/мягкий маркер. Окончательную модель FK см. в **13.2 «Модель данных»**.

> **Краевой случай (отметить для согласования).** При TTL access = 15 мин остаётся узкое окно, когда удалённый пользователь ещё держит валидный access. Для нашего масштаба это приемлемо; если заказчик потребует строго мгновенный отзыв и для access — добавляется проверка `jti`/версии пользователя по БД (это мы и так делаем через `get_active_by_id`, поэтому фактическое окно близко к нулю: учётка уже удалена и шаг 2 отсекает запрос). Поэтому окно касается лишь сценария «учётка ещё есть, но e-mail убран из реестра» — закрывается тем, что `is_listed` проверяется отдельно.

#### 13.3.4. Регистрация и сброс пароля

**Регистрация** (`POST /auth/register`) — реализует логику «сначала доступ, потом самостоятельная регистрация» (раздел 3.2–3.3):

```
1. вход: email, password
2. registry_repo.is_listed(email)?
     НЕТ → 403, тело ровно: {"detail": "Извините, у вас нет доступа к сервису."}
            (никаких контактов, подсказок, переадресации — раздел 2/3.3)
     ДА  → проверить, что учётка ещё не создана (иначе 409 «уже зарегистрирован»)
         → создать user(password_hash=argon2id(password)), связать с записью реестра
         → выдать пару токенов
```

> **Проектное решение (защита от перебора реестра).** Фраза отказа отдаётся **дословно** по ТЗ. Чтобы белый список нельзя было «просеять» перебором e-mail, на `/auth/register` и `/auth/password-reset/request` ставится rate-limit по IP (например, через `slowapi`, лимиты в конфиг). Это не меняет видимый текст, требуемый ТЗ.

**Сброс пароля** (раздел 3.6) — через одноразовый токен с TTL:

| Эндпойнт | Поведение |
|---|---|
| `POST /auth/password-reset/request` | вход `email`. Генерируется случайный токен (≥32 байт), в БД хранится его **хэш** + `user_id` + `expires_at` (TTL из конфига, по умолч. **60 мин**) + `used_at=NULL`. Ссылка с **сырым** токеном уходит на e-mail (канал — обязательный, см. 13.4). **Ответ всегда одинаковый и нейтральный** (200 «если адрес зарегистрирован, письмо отправлено») — чтобы не раскрывать, кто в реестре. |
| `POST /auth/password-reset/confirm` | вход `token`, `new_password`. Проверка: токен существует, не `used`, не истёк, e-mail всё ещё в реестре. Успех → `password_hash` обновляется, токен помечается `used_at`, **все refresh-токены пользователя отзываются** (вынуждаем перелогиниться). |

Таблица `password_reset_tokens(token_hash, user_id, expires_at, used_at)`. Токены одноразовые и протухают; повторный запрос инвалидирует прежние неиспользованные (опционально).

#### 13.3.5. Привилегия администратора (директор)

`is_admin` — **отдельный булев флаг на учётной записи `user`**, ортогональный ролям по задачам (раздел 3.8). Один человек одновременно: обычный пользователь (ставит/исполняет/наблюдает задачи) **и** админ (доступ к админ-панели). Это не «роль-аккаунт» в смысле задач — оно не пересекается с per-task ролями.

- Первичный админ создаётся **сид-скриптом** из переменных окружения (`ADMIN_EMAIL`, `ADMIN_PASSWORD`) при развёртывании; его e-mail при сиде заносится и в реестр (см. **13.2 «Модель данных» / 13.7 «Развёртывание»**).
- Доступ к `/admin/*` закрывается отдельной зависимостью:

```python
async def require_admin(user: CurrentUser = Depends(get_current_user)) -> CurrentUser:
    # перепроверяем по БД, а не доверяем claim из JWT (флаг мог быть снят)
    if not await users_repo.is_admin(db, user.id):
        raise HTTPException(403, "ADMIN_REQUIRED")
    return user
```

> **Проектное решение.** Решение об админ-доступе **перепроверяется по БД** (`users_repo.is_admin`), а не берётся только из claim `is_admin` в JWT, — чтобы снятие привилегии действовало без ожидания истечения access-токена. claim в JWT используется лишь как подсказка для UI.

#### 13.3.6. Серверная авторизация на каждом действии: per-task роли

Роли **вычисляются на лету для конкретной задачи** (разделы 4 и 6): постановщик = `author_id`, исполнитель = `assignee_id`, наблюдатель = `user_id ∈ task_observers`. У пользователя нет глобальной роли. Механизм:

**Шаг 1 — единая dependency `get_task_context`** загружает задачу и вычисляет роль текущего пользователя по ней. Любой роутер задачи получает уже готовый `TaskContext` (нельзя «забыть» проверку):

```python
@dataclass
class TaskContext:
    task: Task
    user: CurrentUser
    role: TaskRole            # AUTHOR | ASSIGNEE | OBSERVER | NONE

async def get_task_context(
    task_id: int,
    user: CurrentUser = Depends(get_current_user),
    db: AsyncSession = Depends(get_db),
) -> TaskContext:
    task = await tasks_repo.get_with_observers(db, task_id)
    if task is None:
        raise HTTPException(404, "TASK_NOT_FOUND")
    role = domain.roles.role_of(user.id, task)   # чистая функция домена
    if role is TaskRole.NONE:
        raise HTTPException(404, "TASK_NOT_FOUND")  # 404, не 403 — не раскрываем существование
    return TaskContext(task=task, user=user, role=role)
```

`role_of` — чистая доменная функция (тестируется без БД):

```python
def role_of(user_id: int, task: Task) -> TaskRole:
    if task.author_id   == user_id: return TaskRole.AUTHOR
    if task.assignee_id == user_id: return TaskRole.ASSIGNEE
    if user_id in task.observer_ids: return TaskRole.OBSERVER
    return TaskRole.NONE
```

> **Проектное решение (видимость).** Для задачи, к которой нет доступа, возвращаем **404, а не 403** — чтобы не подтверждать существование чужих задач (согласуется с приватностью наблюдателя и областью поиска по 6-значному ID из раздела 7.5). Поиск по коду (`/tasks/search`) фильтруется тем же `role_of`.

**Шаг 2 — декларативная политика прав** в `domain/permissions.py` как явная матрица «действие → допустимые роли». Это машинная копия таблицы раздела 6, единственный источник истины:

```python
class Action(StrEnum):
    EDIT_FIELDS=...; DELETE_TASK=...; CHANGE_STATUS=...
    ADD_REPORT=...; MARK_READY=...; VIEW=...; EXPORT=...; ADD_ATTACHMENT_TASK=...

ALLOWED: dict[Action, set[TaskRole]] = {
    Action.VIEW:                {AUTHOR, ASSIGNEE, OBSERVER},
    Action.EDIT_FIELDS:         {AUTHOR},
    Action.DELETE_TASK:         {AUTHOR},
    Action.CHANGE_STATUS:       {AUTHOR},
    Action.ADD_ATTACHMENT_TASK: {AUTHOR},        # к задаче — постановщик
    Action.ADD_REPORT:          {ASSIGNEE},      # отчёт/вложение к отчёту — исполнитель
    Action.MARK_READY:          {ASSIGNEE},
    Action.EXPORT:              {AUTHOR, ASSIGNEE, OBSERVER},  # каждый — свой список
}

def authorize(action: Action, ctx: TaskContext) -> None:
    if ctx.role not in ALLOWED[action]:
        raise HTTPException(403, "FORBIDDEN")
```

**Шаг 3 — вызов в сервисе перед мутацией.** Роутер тонкий; решение принимается централизованно:

```python
# routers/tasks.py
@router.patch("/tasks/{task_id}/status")
async def change_status(body: StatusIn, ctx=Depends(get_task_context), db=...):
    return await task_service.change_status(ctx, body.status, db)

# services/task_service.py
async def change_status(ctx, new_status, db):
    authorize(Action.CHANGE_STATUS, ctx)          # сервер. проверка прав
    domain.status.validate_transition(ctx.task.status, new_status)
    ...
```

**Таблица «действие → проверка»** (как технически проверяется каждое право; полная семантика — таблица раздела 6):

| Действие (эндпойнт) | Guard / проверка на сервере | Допустимая роль |
|---|---|---|
| Создать задачу `POST /tasks` | `get_current_user`; `author_id := user.id` принудительно (из тела не берётся) | любой залогиненный → станет AUTHOR |
| Изменить поля `PUT /tasks/{id}` | `get_task_context` + `authorize(EDIT_FIELDS)` | AUTHOR |
| Удалить задачу `DELETE /tasks/{id}` | `get_task_context` + `authorize(DELETE_TASK)` | AUTHOR |
| Сменить статус `PATCH /tasks/{id}/status` | `authorize(CHANGE_STATUS)` + `status.validate_transition` | AUTHOR |
| Добавить отчёт `POST /tasks/{id}/report` | `authorize(ADD_REPORT)` | ASSIGNEE |
| Отметить готовность `POST /tasks/{id}/mark-ready` | `authorize(MARK_READY)` | ASSIGNEE |
| Вложение к задаче `POST /tasks/{id}/attachments` (контекст=task) | `authorize(ADD_ATTACHMENT_TASK)` | AUTHOR |
| Вложение к отчёту (контекст=report) | `authorize(ADD_REPORT)` | ASSIGNEE |
| Просмотр `GET /tasks/{id}` | `get_task_context` (роль ≠ NONE) | AUTHOR/ASSIGNEE/OBSERVER |
| Просмотр отчёта | входит в `VIEW` карточки | AUTHOR/ASSIGNEE/OBSERVER(свои) |
| Список `GET /tasks?role=...` | фильтр по `author_id`/`assignee_id`/`observers` = `user.id`; параметр `role` не расширяет видимость | по фактической принадлежности |
| Поиск `GET /tasks/search?code=` | находит, затем `role_of ≠ NONE`, иначе 404 | в пределах видимости |
| Выгрузка `GET /tasks/export` | та же выборка, что и список | по принадлежности |
| Флаг «Просрочена» | **только система** (см. 13.4); нет эндпойнта для пользователя | — (авто) |
| Реестр `*/admin/registry`, `DELETE /admin/users/{id}` | `require_admin` (перепроверка `is_admin` по БД) | ADMIN |

Важный приём против «обхода через API»: **серверные поля назначаются сервером, а не принимаются из тела**. `author_id` всегда = текущий пользователь; статус «Просрочена» недоступен в DTO; смена `assignee_id`/наблюдателей идёт только через `PUT` под `EDIT_FIELDS`.

#### 13.3.7. Серверная валидация инвариантов

Инварианты раздела 5/11 проверяются в `domain/invariants.py` и вызываются из `services/` при создании и любом изменении задачи. Pydantic-схема даёт первый (форматный) барьер, но **доменная проверка обязательна** — чтобы инвариант нельзя было обойти кастомным запросом.

| Инвариант (ТЗ) | Где и как проверяется |
|---|---|
| Исполнитель **ровно один** | `assignee_id: int` (не список, NOT NULL в БД); domain: `assignee_id` обязателен и есть в реестре |
| Наблюдателей **≤ 5** | Pydantic `max_length=5` на списке + domain `len(set(observer_ids)) <= 5`; в БД — проверка/триггер `COUNT ≤ 5` на `task_observers` |
| Наблюдатели **уникальны** и ≠ исполнитель/постановщик (рекоменд.) | domain: дедупликация, валидация пересечений |
| Все участники — **из реестра** | domain проверяет `registry_repo.are_listed(ids)` перед сохранением |
| Название **обязательно** | Pydantic `min_length=1` + domain `not blank` |
| Срок **обязателен** (раздел 5/9) | Pydantic required + domain: валиден режим `date` / `datetime` |
| Статусные переходы | `status.validate_transition` (например, из `Отменена` нельзя в `Выполнена`; точную статус-машину см. 13.2) |
| Сквозной номер без дыр/дублей | repo: атомарный захват из `task_seq` (PostgreSQL `SEQUENCE` или `SELECT … FOR UPDATE` по счётчику) в той же транзакции — детали в 13.2 |
| Уникальный 6-значный код | repo: генерация в `100000–999999`, `UNIQUE`-индекс, повтор при коллизии (раздел 5) — детали в 13.2 |

Стандартизированная обработка ошибок: доменные исключения (`InvariantError`, `PermissionDenied`, `TransitionError`) ловятся одним exception-handler уровня приложения и маппятся в коды `422 / 403 / 409` с предсказуемым телом — роутеры не дублируют эту логику.

### 13.4. Подсистема уведомлений и планировщик

Зона отвечает за два слабо связанных контура: **достоверность флага «Просрочена»** (частый лёгкий процесс) и **гарантированную доставку плановых уведомлений по графику из 4 событий** (суточный фоновый прогон). Каналы доставки спроектированы как подключаемые адаптеры с заделом под push/SMS. Модель пользователей/реестра и сущность задачи описаны в смежных подразделах (см. «Модель данных» и «Аутентификация/реестр»); здесь они используются как данность.

#### 13.4.1. Архитектура каналов доставки (адаптеры)

Канал — это абстракция с единственной обязанностью «доставить готовое сообщение конкретному адресату». Формирование текста, выбор получателей и идемпотентность лежат **выше** канала, в слое оркестрации. Это разделение обязательно: оно позволяет добавить push/SMS, не трогая бизнес-логику графика.

```python
# domain/notifications/channel.py
from dataclasses import dataclass
from enum import Enum

class ChannelKind(str, Enum):
    EMAIL = "email"
    MAX   = "max"
    # PUSH = "push"   # задел, не реализуется
    # SMS  = "sms"    # задел, не реализуется

class DeliveryStatus(str, Enum):
    DELIVERED   = "delivered"     # канал подтвердил приём
    FAILED      = "failed"        # доставка не удалась (best-effort каналам не критично)
    SKIPPED     = "skipped"       # нет адреса/привязки для этого канала

@dataclass(frozen=True)
class Attachment:
    filename: str
    size: int
    content_type: str
    storage_path: str             # путь в файловом хранилище (см. подраздел вложений)
    public_url: str               # ссылка для скачивания (используется при деградации в MAX)

@dataclass(frozen=True)
class Message:
    subject: str                  # тема (для e-mail); MAX игнорирует
    body_text: str                # текст уведомления (рус.)
    body_html: str | None         # html-версия для e-mail
    attachments: list[Attachment]
    task_code: str                # 6-значный ID — для удобства поиска получателем

@dataclass(frozen=True)
class Recipient:
    user_id: int
    email: str                    # обязательный реквизит реестра
    max_ref: str | None           # chat_id/телефон/идентификатор бота MAX; None → MAX недоступен

@dataclass(frozen=True)
class DeliveryResult:
    channel: ChannelKind
    status: DeliveryStatus
    detail: str | None            # код/текст ошибки провайдера для журнала

class NotificationChannel(Protocol):
    kind: ChannelKind
    def can_send(self, r: Recipient) -> bool: ...          # есть ли адрес/привязка
    def send(self, r: Recipient, m: Message) -> DeliveryResult: ...
```

**Проектное решение.** Канал возвращает `DeliveryResult`, а не бросает исключение на best-effort-ошибках. Только **`EmailChannel`** считается *гарантированным*: его сбой логируется и помечается для повторной попытки в следующем прогоне (через журнал, см. 13.4.5). **`MaxChannel`** — best-effort: любой сбой/`SKIPPED` не блокирует событие и не считается «недоставленным» для целей идемпотентности.

| Реализация | Статус | Транспорт | Поведение при сбое |
|---|---|---|---|
| `EmailChannel` | обязательный, гарантированный | SMTP (`aiosmtplib`), параметры из `.env` | помечается для ретрая, событие не закрывается успешно |
| `MaxChannel` | best-effort | HTTP API бота MAX | проглатывается, событие закрывается по факту успеха почты |
| `PushChannel`/`SmsChannel` | **только задел** | — | классы не реализуются; интерфейс готов |

##### Порядок доставки и проблемные вложения в MAX

Для **события №1 (постановка)** заказчик зафиксировал порядок «сначала почта, потом MAX». Оркестратор вызывает каналы строго последовательно в порядке `[EMAIL, MAX]`:

```python
CHANNEL_ORDER = [ChannelKind.EMAIL, ChannelKind.MAX]
```

`EmailChannel` отправляет **все** вложения как есть (почта — гарантированный канал для файлов). `MaxChannel` применяет деградацию: MAX принимает не все типы (подтверждено — HTML не проходит). Логика канала MAX:

```python
MAX_BLOCKED_EXT = {".html", ".htm", ".svg", ...}      # в конфиг
MAX_MAX_FILE_SIZE = 20 * 1024 * 1024                  # лимит MAX, в конфиг

def _max_payload(m: Message) -> tuple[str, list[Attachment]]:
    sendable, degraded = [], []
    for a in m.attachments:
        ext = Path(a.filename).suffix.lower()
        if ext in MAX_BLOCKED_EXT or a.size > MAX_MAX_FILE_SIZE:
            degraded.append(a)
        else:
            sendable.append(a)
    body = m.body_text
    if degraded:
        links = "\n".join(f"• {a.filename}: {a.public_url}" for a in degraded)
        body += ("\n\nЧасть вложений недоступна в MAX, "
                 f"они отправлены на почту и доступны по ссылке:\n{links}")
    return body, sendable
```

**Проектное решение.** Проблемное вложение в MAX → (а) текстовая **ссылка** на скачивание в теле MAX-сообщения **и** (б) гарантированное **дублирование почтой** (почта и так шлёт все файлы первой). Это покрывает оба варианта, упомянутых заказчиком («через ссылку», «через почту тоже неплохо»), без отдельного флага конфигурации.

#### 13.4.2. Разделение двух процессов по частоте

Контуры разнесены намеренно: достоверность флага не должна зависеть от тяжёлого суточного прогона, а суточная рассылка не должна гоняться каждые 5 минут.

```
┌──────────────────────────────────────────────────────────────────────┐
│  Процесс A: ФЛАГ «ПРОСРОЧЕНА» (лёгкий, частый + по факту чтения)        │
│                                                                        │
│  A1. "По факту" при чтении: is_overdue = now >= due_moment(task)       │
│      вычисляется в репозитории/сериализаторе на каждый GET.            │
│  A2. Лёгкий проход overdue_sweep() каждые 5–15 мин:                    │
│      UPDATE tasks SET is_overdue = true, overdue_since = now()       │
│      WHERE status='В работе' AND is_overdue=false                    │
│        AND now() >= due_moment;                                        │
│      → фиксирует ФАКТ просрочки в БД (для отчётности и бейджа),        │
│        не шлёт писем.                                                   │
└──────────────────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────────────────┐
│  Процесс B: ПЛАНОВЫЕ УВЕДОМЛЕНИЯ (тяжёлый, раз в сутки)                 │
│                                                                        │
│  daily_run() в SCHED_DAILY_TIME (по умолч. 09:00 Europe/Moscow):       │
│  события 1–4, выбор получателей, рендер текста, доставка по каналам,   │
│  запись в журнал идемпотентности.                                      │
└──────────────────────────────────────────────────────────────────────┘
```

**Почему именно так.** Флаг «Просрочена» обязан быть достоверен в режиме «дата+время» (срок мог истечь в 14:37) — ждать прогона 09:00 нельзя. Поэтому:
- **A1 («по факту»)** — единственный источник истины для **отображения** (бейдж, фильтры, выгрузка): даже если воркер лежит, открытая карточка покажет корректный флаг, потому что `is_overdue` пересчитывается из `due_moment` и `now()`.
- **A2 (частый sweep)** — материализует факт в БД: проставляет `is_overdue=true` и неизменяемую метку `overdue_since`, чтобы факт сохранялся в отчётности независимо от дальнейших чтений (раздел 5: «однажды возникнув, флаг не обнуляется автоматически»).

> Разделение A1/A2 устраняет противоречие ТЗ: флаг — это и «вычислять по факту», и «материализованный факт». A1 даёт мгновенную достоверность для UI, A2 — персистентность для отчётности.

`due_moment(task)` — единая функция, используемая обоими процессами и графиком (раздел 5):

```python
def due_moment(task) -> datetime:                  # всегда aware, в TZ организации
    if task.due_mode == "datetime":
        return task.due_at                          # точный момент
    # режим "только дата": просрочка с началом СЛЕДУЮЩИХ суток
    return (task.due_date + timedelta(days=1)).start_of_day(tz=ORG_TZ)
```

#### 13.4.3. График из 4 событий: получатели, тайминг, идемпотентность

`due_date_local` — календарная дата срока в TZ организации: для режима «только дата» это `due_date`; для «дата+время» — `due_at.date()` в TZ организации. `run_date` — дата текущего суточного прогона.

| № | Событие | Получатели | Условие срабатывания в прогоне `run_date` | Текст | Ключ идемпотентности |
|---|---|---|---|---|---|
| 1 | День постановки | **исполнитель** + **все наблюдатели** (≤5), разный текст | `run_date == created_at_local.date()` **или** «догоняющий» при первом прогоне после создания (см. краевые) | исп.: «Вам поставлена задача №… (вы исполнитель)»; набл.: «Вы назначены наблюдателем по задаче №…»; с текстом и вложениями | `(task_id, "assigned", recipient_id)` — **без даты и версии срока**: одноразово на участника |
| 2 | За сутки до срока | **только исполнитель** | `run_date == due_date_local − 1 день` и задача открыта | «Завтра истекает срок исполнения поручения №…» | `(task_id, "due_tomorrow", due_version, run_date)` |
| 3 | В день срока | **только исполнитель** | `run_date == due_date_local` и задача открыта | «Срок вашей задачи №… истекает сегодня» | `(task_id, "due_today", due_version, run_date)` |
| 4 | После срока, ежедневно | **только исполнитель** | `run_date > due_date_local` и задача открыта (`В работе`) | «Задача №… просрочена. Срок истёк …» | `(task_id, "overdue_daily", due_version, run_date)` |

Пояснения:
- **Событие 1** — единственное для **наблюдателей**; его ключ не содержит даты и версии срока, потому что назначение участником одноразово (изменение срока на него не влияет; см. 13.4.4).
- **События 2–4** привязаны к `due_version` — целочисленному счётчику версии срока (см. 13.4.4). При смене срока ключи становятся другими, и напоминания корректно срабатывают заново по новому сроку.
- **«Задача открыта»** = `status == 'В работе'`. Для `Выполнена`/`Отменена` события 2–4 не шлются (раздел 9: ежедневные напоминания прекращаются после закрытия).
- Все условия используют **календарные даты в TZ организации**, поэтому единый суточный прогон корректно обслуживает оба режима срока (раздел 9).

##### Структура журнала идемпотентности

```sql
CREATE TABLE notification_log (
    id              BIGSERIAL PRIMARY KEY,
    task_id         BIGINT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
    event_type      TEXT   NOT NULL,        -- assigned|due_tomorrow|due_today|overdue_daily
    recipient_id    BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    due_version     INT    NULL,            -- NULL для assigned; счётчик версии срока для 2–4
    run_date        DATE   NULL,            -- NULL для assigned; дата прогона для 2–4
    -- результаты по каналам (для гарантированности почты и ретраев)
    email_status    TEXT   NOT NULL,        -- delivered|failed|skipped
    max_status      TEXT   NOT NULL,
    detail          TEXT   NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Ключ идемпотентности 2–4: один тип в день на задачу/версию/получателя
CREATE UNIQUE INDEX uq_notif_dated ON notification_log
    (task_id, event_type, recipient_id, due_version, run_date)
    WHERE run_date IS NOT NULL;

-- Ключ идемпотентности события 1: одноразово на участника
CREATE UNIQUE INDEX uq_notif_assigned ON notification_log
    (task_id, event_type, recipient_id)
    WHERE event_type = 'assigned';
```

Идемпотентность обеспечивается **уникальным индексом + INSERT ... ON CONFLICT DO NOTHING**, а не проверкой «есть ли запись» до отправки. Это устраняет гонки при перезапуске воркера в середине прогона: запись-«бронь» вставляется **перед** доставкой; если строка уже есть — событие в этот день/версию уже обработано, пропускаем. Так перезапуск не порождает дублей (раздел 9, 11).

**Гарантированность почты при сбое.** Если `email_status = failed`, строка журнала остаётся, но при следующем прогоне такой строке разрешён повтор только по почте: запрос на повтор отбирает записи `email_status='failed' AND run_date = today` и пытается доставить заново, не создавая новой брони. MAX-сбои не ретраятся (best-effort).

#### 13.4.4. Переоценка при изменении срока постановщиком

Постановщик вправе менять срок (раздел 6). Версионирование срока на стороне задачи (поле задаётся в смежном подразделе модели данных — здесь фиксируется требование):

```sql
ALTER TABLE tasks ADD COLUMN due_version INT NOT NULL DEFAULT 1;
```

При сохранении нового срока (`PUT /tasks/{id}` меняет `due_*`) выполняется транзакционная переоценка:

```python
def on_due_changed(task, new_due):
    task.set_due(new_due)
    task.due_version += 1                              # инвалидирует ключи 2–4
    new_moment = due_moment(task)
    if datetime.now(ORG_TZ) < new_moment:
        # новый срок ещё не наступил → факт просрочки снимаем
        task.is_overdue  = False
        task.overdue_since = None
    # если новый срок уже в прошлом — флаг проставит ближайший sweep (A2)
```

- **Снятие флага.** Раздел 9 прямо требует снимать «Просрочена», если новый срок ещё не наступил. Это исключение из правила «флаг не обнуляется автоматически» (раздел 5): обнуление здесь — следствие **явного действия постановщика**, а не авто-сброса.
- **Привязка ключей к версии.** Инкремент `due_version` означает, что ключи событий 2–4 для старой версии больше не совпадут с новыми; напоминания «за сутки»/«в день срока»/«ежедневно» гарантированно сработают заново по новому сроку. Старые записи журнала остаются как история.
- **Событие 1 не повторяется** при смене срока: его ключ не содержит `due_version` — участники уже извещены о постановке. Опциональное best-effort уведомление «срок изменён» отправляется отдельным типом (вне обязательного MVP).

##### Краевые случаи

| Случай | Поведение |
|---|---|
| Задача создана в день срока | Событие 1 уходит; событие 2 («за сутки») **пропускается** (его день уже в прошлом); событие 3 («в день срока») уходит в этот же прогон |
| Задача создана **после** срока | Событие 1 уходит; 2 и 3 пропускаются; начинается ежедневное событие 4; флаг проставит ближайший sweep |
| Воркер не запускался сутки (простой) | При следующем прогоне событие 1 «догоняет» по `assigned`-ключу (одноразовый, не привязан к дате); пропущенные 2/3 за прошлый день **не отсылаются задним числом** (run_date уже не совпадает); событие 4 отправится за текущий день |
| Новый срок в прошлом | Флаг проставится ближайшим sweep; события 2/3 для новой версии пропустятся, начнётся 4 |
| Срок отодвинут в будущее у просроченной задачи | Флаг снят (`now < new_moment`); `due_version++`; события 2–4 отработают заново по новому сроку |

#### 13.4.5. Псевдокод суточного прогона

```python
def daily_run():
    """Запускается раз в сутки в SCHED_DAILY_TIME (TZ организации)."""
    run_date = today(ORG_TZ)

    # Сначала повторная попытка по почте для вчера-сегодня не доставленных (гарантированность)
    retry_failed_email(run_date)

    for task in iter_open_and_recently_closed_tasks():   # 'В работе' + закрытые в этот день для события 1
        dl   = due_date_local(task)
        recs = build_recipients(task)                     # из реестра: e-mail обяз., max_ref опц.

        # СОБЫТИЕ 1 — постановка (исполнитель + наблюдатели), одноразово
        if is_assignment_pending(task, run_date):
            emit(task, "assigned", task.assignee,  tmpl_assigned_executor(task),
                 due_version=None, run_date=None, order=[EMAIL, MAX])     # «сначала почта, потом MAX»
            for obs in task.observers:                                   # ≤5
                emit(task, "assigned", obs, tmpl_assigned_observer(task),
                     due_version=None, run_date=None, order=[EMAIL, MAX])

        if task.status != "В работе":
            continue                                       # 2–4 только для открытых

        # СОБЫТИЕ 2 — за сутки
        if run_date == dl - 1*day:
            emit(task, "due_tomorrow", task.assignee, tmpl_due_tomorrow(task),
                 due_version=task.due_version, run_date=run_date)

        # СОБЫТИЕ 3 — в день срока
        elif run_date == dl:
            emit(task, "due_today", task.assignee, tmpl_due_today(task),
                 due_version=task.due_version, run_date=run_date)

        # СОБЫТИЕ 4 — ежедневно после срока
        elif run_date > dl:
            emit(task, "overdue_daily", task.assignee, tmpl_overdue(task),
                 due_version=task.due_version, run_date=run_date)


def emit(task, event_type, recipient, message, *, due_version, run_date,
         order=(EMAIL, MAX)):
    # 1) БРОНЬ в журнале ДО отправки — гарантия идемпотентности при перезапуске
    inserted = log_insert_or_ignore(task.id, event_type, recipient.id,
                                    due_version, run_date)
    if not inserted:
        return                                              # уже отправлено сегодня/в эту версию

    # 2) Доставка по каналам строго в заданном порядке
    results = {}
    for kind in order:
        ch = channel(kind)
        results[kind] = ch.send(recipient, message) if ch.can_send(recipient) \
                        else DeliveryResult(kind, SKIPPED, "no address")

    # 3) Фиксация результатов; почта-failed останется кандидатом на ретрай
    log_update_results(task.id, event_type, recipient.id, due_version, run_date,
                       email=results[EMAIL], max=results.get(MAX))
```

Ключевые свойства: бронь-перед-отправкой даёт строгую идемпотентность (≤1 уведомление каждого типа в день, перезапуск без дублей); порядок `[EMAIL, MAX]` реализует требование «сначала почта»; почта-`failed` подбирается `retry_failed_email` в следующих прогонах.

#### 13.4.6. Технологический выбор планировщика: APScheduler в процессе воркера

**Проектное решение — APScheduler внутри отдельного контейнера-воркера, без внешнего брокера.** Обоснование для скромного школьного self-host:

| Критерий | APScheduler в воркере | cron в контейнере |
|---|---|---|
| Зависимости | только Python (уже в стеке) | системный crond в образе |
| Доступ к коду/ORM | прямой импорт моделей и каналов | отдельный entrypoint-скрипт на каждый прогон |
| Два расписания (sweep 5–15 мин + daily) | оба джоба в одном процессе | две crontab-строки + дубль bootstrap |
| TZ организации | нативно (`timezone=Europe/Moscow`) | зависит от TZ контейнера, легко ошибиться |
| Идемпотентность при перекрытии | `max_instances=1`, `coalesce=True` | нужна внешняя блокировка от наложений |
| Конфиг времени из `.env` | прямо в коде из переменной | редактирование crontab, нет валидации |
| Внешний брокер (Celery/Redis) | **не нужен** (раздел 10 — простота) | не нужен |

APScheduler (`AsyncIOScheduler`) поднимается в `worker`-контейнере docker-compose и регистрирует два джоба:

```python
scheduler.add_job(overdue_sweep, IntervalTrigger(minutes=OVERDUE_SWEEP_MINUTES),  # 5–15
                  id="overdue_sweep", max_instances=1, coalesce=True)
scheduler.add_job(daily_run, CronTrigger(hour=H, minute=M, timezone=ORG_TZ),
                  id="daily_run", max_instances=1, coalesce=True, misfire_grace_time=3600)
```

`coalesce=True` + `misfire_grace_time` гарантируют: если воркер был недоступен в момент 09:00, прогон выполнится один раз при восстановлении (а не пропустится и не задвоится). `max_instances=1` исключает наложение sweep на самого себя. Персистентность расписания в БД не требуется — джобы статичны и пересоздаются при старте; источник истины по «что уже отправлено» — `notification_log`, а не состояние планировщика.

**Конфигурация (в `.env.example`, смежный подраздел деплоя):** `SCHED_DAILY_TIME=09:00`, `ORG_TZ=Europe/Moscow`, `OVERDUE_SWEEP_MINUTES=10`, `MAX_BLOCKED_EXT`, `MAX_MAX_FILE_SIZE`, SMTP-параметры, параметры бота MAX.

#### 13.4.7. Задел под push/SMS (без переписывания)

- Новый канал = новый класс, реализующий `NotificationChannel`, и регистрация в реестре каналов; оркестратор, граф событий и журнал **не меняются**.
- `ChannelKind` расширяется значениями `PUSH`/`SMS`; `notification_log` уже хранит результаты по каналам — добавятся столбцы `push_status`/`sms_status` миграцией без изменения логики идемпотентности.
- `Recipient` пополнится опциональными `push_token`/`phone`; `can_send` каждого канала сам решает доступность — отсутствующая привязка даёт `SKIPPED`, как сейчас у MAX.
- Порядок доставки события 1 («сначала почта») задаётся списком `CHANNEL_ORDER` — добавление каналов меняет только этот список и конфиг, не код графика.

### 13.5. REST API-контракт и файловое хранилище вложений

Подраздел описывает **внешний контракт backend-сервиса** (FastAPI) и **подсистему хранения вложений**. Модель данных, миграции и счётчики (сквозной номер, 6-значный ID) — см. смежный подраздел «Модель данных и БД»; правила планировщика и тексты уведомлений — см. подраздел «Уведомления и планировщик». Здесь фиксируются только сетевой интерфейс, проверки прав на каждом запросе и работа с файлами.

#### 13.5.1. Общие соглашения контракта

- **Базовый префикс:** `/api/v1`. Версионирование в пути — задел на эволюцию контракта без слома клиентов (push/SMS, «проект» добавятся как новые поля/эндпойнты, не ломая v1).
- **Формат:** JSON (`Content-Type: application/json; charset=utf-8`), кроме загрузки файлов (`multipart/form-data`) и печатной выгрузки (`text/html`).
- **Аутентификация:** `Authorization: Bearer <access_token>` (JWT). Логин — e-mail, отдельного логина нет (раздел 3). Детали жизненного цикла токена (срок, refresh) — см. подраздел «Аутентификация и сессии»; контракт лишь требует Bearer-токен и серверную проверку «пользователь всё ещё в реестре» **на каждом** запросе (п. 12 раздела 3).
- **Идентификаторы в путях:** `{id}` задачи — это **внутренний UUID/PK задачи**, а не сквозной номер и не 6-значный код. Сквозной номер и 6-значный код — поля в теле ответа. Это разделяет «ссылку на ресурс» и «человекочитаемые идентификаторы» и не даёт перечислять задачи подбором номера.
- **Локаль:** все человекочитаемые сообщения — на русском (`Accept-Language` игнорируется, язык один — раздел 1).
- **Время:** все метки времени в ответах — ISO-8601 UTC (`...Z`); срок задачи хранит признак режима (`due_mode: "date" | "datetime"`) — см. подраздел «Модель данных». Клиент отображает в `Europe/Moscow`.
- **Идемпотентность мутаций:** на «небезопасных» действиях, повтор которых нежелателен (`mark-ready`, смена статуса), сервер опирается на текущее состояние (см. коды `409`), а не на клиентский повтор.

**Проектное решение.** Префикс `/api/v1` и UUID в путях — предложение разработчика; директором не оговаривалось. Альтернатива (плоские пути без версии, номер в URL) проще, но хуже расширяется и небезопаснее по перечислению. Рекомендуется версионирование.

#### 13.5.2. Единая модель ошибок

Все ошибки возвращаются **единым телом** (включая ошибки валидации FastAPI/Pydantic — переопределяется обработчик `RequestValidationError`):

```jsonc
{
  "error": {
    "code": "TASK_NOT_FOUND",        // машинный код (UPPER_SNAKE), стабилен
    "message": "Задача не найдена.",  // локализованное сообщение (рус.)
    "details": [                      // опционально, для 422 — по полям
      { "field": "observers", "message": "Не более 5 наблюдателей." }
    ]
  }
}
```

- `code` — **контрактная константа**, по ней фронтенд ветвит логику (не по тексту). Текст можно менять без слома клиента.
- HTTP-статус + `code` согласованы. Каталог кодов:

| HTTP | code | Когда | Сообщение (рус.) |
|---|---|---|---|
| 400 | `BAD_REQUEST` | Неразбираемое тело/параметр | «Некорректный запрос.» |
| 401 | `UNAUTHENTICATED` | Нет/невалиден токен, пользователь не вошёл | «Требуется вход в систему.» |
| 401 | `REGISTRY_ACCESS_REVOKED` | Токен валиден, но e-mail удалён из реестра | «Доступ к сервису прекращён.» |
| 403 | `FORBIDDEN_ROLE` | Аутентифицирован, но роль на задаче не даёт действия | «Недостаточно прав для этого действия.» |
| 403 | `FORBIDDEN_ADMIN` | Не админ для админ-эндпойнта | «Действие доступно только администратору.» |
| 404 | `TASK_NOT_FOUND` | Задача не существует **или** вне области видимости (см. ниже) | «Задача не найдена.» |
| 404 | `USER_NOT_FOUND` | Нет такого пользователя/записи реестра | «Пользователь не найден.» |
| 409 | `STATUS_CONFLICT` | Недопустимый переход статуса / действие над закрытой задачей | «Действие невозможно в текущем статусе задачи.» |
| 409 | `ALREADY_READY` | Повторная отметка готовности | «Готовность уже отмечена.» |
| 409 | `EMAIL_ALREADY_REGISTERED` | Повторная регистрация e-mail | «Этот e-mail уже зарегистрирован.» |
| 422 | `VALIDATION_ERROR` | Бизнес-валидация полей (см. `details`) | «Проверьте правильность заполнения полей.» |
| 413 | `FILE_TOO_LARGE` | Превышен лимит размера файла | «Файл превышает допустимый размер.» |
| 409 | `ATTACHMENTS_LIMIT` | Превышен лимит количества вложений | «Превышено допустимое число вложений.» |
| 415 | `UNSUPPORTED_FILENAME` | Пустое/небезопасное имя файла | «Недопустимое имя файла.» |
| 429 | `RATE_LIMITED` | Антибрутфорс на `/auth/*` | «Слишком много попыток, повторите позже.» |
| 500 | `INTERNAL` | Непредвиденная ошибка | «Внутренняя ошибка сервиса.» |

**Особый случай отказа в регистрации (раздел 3, п. 3).** E-mail вне реестра при `POST /auth/register` — это **не** ошибка авторизации в обычном смысле, а строго фиксированный отказ. Возвращается `403` с телом:
```jsonc
{ "error": { "code": "EMAIL_NOT_IN_REGISTRY",
             "message": "Извините, у вас нет доступа к сервису." } }
```
Сообщение — **дословно из ТЗ**, без контактов и подсказок (анти-рамка раздела 2). Фронтенд показывает `message` как есть.

**Принцип «404 вместо 403» для скрытых ресурсов.** Если пользователь запрашивает существующую задачу, к которой у него **нет** роли (постановщик/исполнитель/наблюдатель), сервер отвечает `404 TASK_NOT_FOUND`, а не `403`. Это не раскрывает факт существования чужой задачи и согласуется с приватностью наблюдателя (раздел 4) и областью поиска (раздел 7, п. 5). `403 FORBIDDEN_ROLE` используется только когда пользователь **видит** задачу (есть роль), но конкретное действие его роли запрещено (например, исполнитель пытается сменить статус).

#### 13.5.3. Область видимости (scope) — единое правило

Вводится серверная функция `visible_tasks(user)` — фундамент списков, карточки, поиска и экспорта. Пользователь «видит» задачу, если он по ней **постановщик, ИЛИ исполнитель, ИЛИ один из наблюдателей**. Администратор-привилегия **не расширяет** видимость задач (админка — только реестр; раздел 3): директор видит чужие задачи лишь там, где сам участвует.

```python
def visible_task_ids(user_id) -> set[UUID]:
    # author_id == user OR assignee_id == user OR user in observers
    ...
```

Все читающие эндпойнты (`GET /tasks`, `GET /tasks/{id}`, `GET /tasks/search`, `GET /tasks/export`) фильтруются через это правило **на сервере**. Параметр `?role=` лишь сужает уже видимое подмножество под конкретную вкладку, но **не может расширить** его (раздел 11: ограничения нельзя обойти через API).

Ниже — метод, путь, краткое тело, статусы, требуемая проверка. Поля тел даны сокращённо; полные схемы — Pydantic-модели в коде.

##### Аутентификация и сброс пароля

```
POST /api/v1/auth/register
  body:  { "email": str, "password": str }
  200:   { "user": { "id", "email", "is_admin" } }   // + сессия/токен — см. подраздел «Аутентификация»
  403:   EMAIL_NOT_IN_REGISTRY  (e-mail вне реестра — дословная фраза отказа)
  409:   EMAIL_ALREADY_REGISTERED
  422:   VALIDATION_ERROR (слабый пароль/невалидный e-mail)
  Проверка: e-mail ∈ реестр И ещё не зарегистрирован.

POST /api/v1/auth/login
  body:  { "email": str, "password": str }
  200:   { "access_token": str, "token_type": "bearer", "user": {...} }
  401:   UNAUTHENTICATED (неверная пара)  | REGISTRY_ACCESS_REVOKED (удалён из реестра)
  429:   RATE_LIMITED
  Проверка: пароль верен И e-mail всё ещё в реестре.

POST /api/v1/auth/password-reset/request
  body:  { "email": str }
  200:   { "ok": true }     // ВСЕГДА 200 — не раскрываем наличие e-mail в системе
  Эффект: если e-mail зарегистрирован И в реестре — выслать токен-ссылку (TTL, раздел 3, п. 6).

POST /api/v1/auth/password-reset/confirm
  body:  { "token": str, "new_password": str }
  200:   { "ok": true }
  400:   BAD_REQUEST (токен истёк/использован/неверен)
  422:   VALIDATION_ERROR (слабый пароль)
```

**Проектное решение.** `password-reset/request` отвечает `200` независимо от наличия адреса — стандартная защита от перечисления пользователей. Это согласуется с духом раздела 2 («не показывать, как добавиться»).

##### Администрирование реестра (только `is_admin`)

```
GET    /api/v1/admin/registry?query=&page=&page_size=
  200:   { "items": [ { "id", "email", "max_contact"?, "registered": bool, "is_admin": bool } ], "total" }
  403:   FORBIDDEN_ADMIN
  Эффект: список записей реестра. registered=true, если владелец уже завёл пароль.

POST   /api/v1/admin/registry
  body:  { "email": str, "max_contact": str?, "is_admin": bool? }
  201:   { "id", "email", ... }
  409:   EMAIL_ALREADY_REGISTERED (e-mail уже в реестре)
  422:   VALIDATION_ERROR
  Эффект: внести e-mail в белый список (владелец затем сам регистрируется — раздел 3).

PUT    /api/v1/admin/registry/{id}
  body:  { "email"? , "max_contact"?, "is_admin"? }
  200:   { ... }   404: USER_NOT_FOUND
  Эффект: обновить запись реестра.

DELETE /api/v1/admin/registry/{id}
  200:   { "ok": true }   404: USER_NOT_FOUND
  Эффект: убрать e-mail из реестра → вход блокируется (раздел 3, п. 11).
  ПРИМЕЧАНИЕ: удаление ИЗ РЕЕСТРА (запись) ≠ удаление АРХИВА — см. следующий эндпойнт.
```

**Проектное решение (разделение двух операций).** ТЗ упоминает «удалить запись реестра» (раздел 3, п. 9) и отдельно «удалить пользователя и весь его архив» (п. 10). В контракте это **два разных эндпойнта**: `DELETE /admin/registry/{id}` (лишить доступа, мягко) и `DELETE /admin/users/{id}` (жёсткое каскадное удаление архива). Это снижает риск необратимой потери данных по неосторожному клику. Согласовать с заказчиком, нужна ли «мягкая» деактивация как отдельный режим.

```
DELETE /api/v1/admin/users/{id}?confirm=true
  200:   { "ok": true, "deleted": { "tasks_as_author": int, "attachments": int } }
  400:   BAD_REQUEST (без confirm=true — серверная страховка под диалог подтверждения)
  403:   FORBIDDEN_ADMIN     404: USER_NOT_FOUND
  Эффект: каскадно удалить пользователя и ВЕСЬ его архив (раздел 3, п. 10):
          - все задачи, где он постановщик (вместе с их вложениями/отчётами/файлами на диске);
          - его учётные данные, токены, привязки.
  В ЧУЖИХ задачах (где он лишь исполнитель/наблюдатель) — по умолчанию НЕ удалять задачу,
  а обезличить участие (см. Проектное решение ниже и подраздел «Модель данных»).
```

**Проектное решение (объём «архива» — на согласование, раздел 3 п. 10).** Каскад затрагивает только задачи, где удаляемый — **постановщик**. Для чужих задач, где он был исполнителем, предлагается перевод ссылки на «пользователь удалён» (обезличивание), чтобы не разрушать целостность чужих задач. Но **исполнитель обязателен и ровно один** (раздел 4) — поэтому осиротевшую чужую задачу нельзя оставить без исполнителя «молча». Варианты на выбор заказчика: (а) перевести такую задачу в статус `Отменена` системой; (б) обязать постановщика переназначить исполнителя (флаг `needs_reassignment`); (в) допустить «технического» псевдо-исполнителя «удалён». Требует решения директора; в контракте поле `deleted.tasks_as_author` уже возвращается для аудита.

##### Задачи (CRUD)

```

  201:   { ...полная карточка..., "seq_no": int, "code": "NNNNNN", "status": "В работе" }
  422:   VALIDATION_ERROR (нет title/due; >5 наблюдателей; assignee отсутствует/не в реестре;

  Кто:   любой с ролью на задаче. Наблюдатель видит карточку только своих задач.

PUT    /api/v1/tasks/{id}
  body:  любые редактируемые поля (title, description, due_at, due_mode, assignee_id, observer_ids, links)
  200:   { обновлённая карточка }
  403:   FORBIDDEN_ROLE (есть доступ, но не постановщик)
  404:   TASK_NOT_FOUND     422: VALIDATION_ERROR
  Кто:   ТОЛЬКО постановщик задачи (раздел 6).
  Эффект: при смене due_at — пересчёт флага «Просрочена» и сброс ключей напоминаний
          (см. подраздел «Уведомления», раздел 9 п. об изменении срока).

DELETE /api/v1/tasks/{id}
  200:   { "ok": true }
  403:   FORBIDDEN_ROLE     404: TASK_NOT_FOUND
  Кто:   ТОЛЬКО постановщик. Эффект: удалить задачу и её вложения с диска.

PATCH  /api/v1/tasks/{id}/status
  body:  { "status": "Выполнена" | "Отменена" | "В работе" }
  200:   { "id", "status" }
  403:   FORBIDDEN_ROLE     404: TASK_NOT_FOUND
  409:   STATUS_CONFLICT (недопустимый переход)
  Кто:   ТОЛЬКО постановщик (раздел 5, 6).
  Эффект: смена статуса НЕ трогает флаг is_overdue (просрочка — отдельный факт, раздел 5).
          Закрытие (Выполнена/Отменена) лишь останавливает ежедневные напоминания.
```

**Проектное решение (`PATCH` для статуса отдельно от `PUT`).** Статус вынесен в отдельный эндпойнт, потому что (а) семантически это узкое действие постановщика с собственными конфликтами (`409`), (б) упрощает аудит и тесты матрицы прав (раздел 11). `PUT /tasks/{id}` **не** меняет статус, даже если поле передано, — оно игнорируется/отвергается (`422`).

#### 13.5.5. Выгрузка под печать

Эндпойнт `…/export?format=print` возвращает **самодостаточную HTML-страницу** (инлайновый `print-CSS`, без внешних ассетов), повторяющую текущую фильтрацию и сортировку списка. Это не «сырые машиночитаемые данные», а готовый к печати/пересылке документ (раздел 8, п. 5):

- Шапка: «Школа № 2090 · Поручения · вкладка ‹роль› · сформировано ДД.ММ.ГГГГ ЧЧ:ММ».
- Таблица с читаемыми колонками: № (seq_no), ID (code), название, описание/срок, исполнитель, постановщик, наблюдатели, статус, бейдж «Просрочена», отчёт о выполнении.
- `@media print`: фиксированная ширина страницы, `table { border-collapse }`, перенос длинного текста (`word-break`), повтор `thead` на каждой странице (`thead { display: table-header-group }`), отсутствие наезжающих элементов.
- Сервер рендерит HTML через Jinja2-шаблон по тем же данным, что `GET /tasks` (общий слой выборки), чтобы экспорт и список не расходились.

**Проектное решение (формат).** По умолчанию — **print-HTML** (просто, без headless-браузера, годится для скромного школьного сервера). PDF (`format=pdf`) — опционально, через WeasyPrint/headless Chromium; включается флагом окружения. Соответствует разделу 8 («предпочтительно простое решение; PDF — опционально»).

#### 13.5.6. Файловое хранилище вложений

**Бэкенд хранения — локальный том** (раздел 10: «файловое хранилище»; S3 — не обязательно). Абстрагируется интерфейсом, чтобы позже подменить на S3-совместимое без переписывания вызывающего кода:

```python
class StorageBackend(Protocol):
    def save(self, task_id: UUID, data: BinaryIO, *, original_name: str) -> StoredFile: ...
    def open(self, stored_key: str) -> BinaryIO: ...
    def delete(self, stored_key: str) -> None: ...

# StoredFile = { stored_key, size, sha256 }
class LocalStorage(StorageBackend): ...   # реализация по умолчанию
```

**Структура путей на диске** (изоляция по задаче, неугадываемые имена):

```
${ATTACHMENTS_ROOT}/
  └── <task_id>/                 # UUID задачи — каталог на задачу
        └── <attachment_uuid>    # UUID вложения = имя файла на диске (БЕЗ исходного имени)
```

- На диске имя файла — **UUID вложения**, без расширения и без исходного имени. Исходное имя (`filename`) и `content_type` хранятся **в БД**, выдаются при скачивании в `Content-Disposition`. Это убирает риск path-traversal и конфликтов имён и не даёт «угадать» файл по URL.
- Каталог `<task_id>` упрощает каскадное удаление (раздел 3, п. 10: при удалении задачи/пользователя сносится весь подкаталог), а также бэкап.

**Лимиты и валидация (раздел 10):** тип файла **не ограничиваем** («можно вложить что угодно»), но вводим разумные пределы — в конфиге (`.env`):

| Параметр | Перем. окружения | Значение по умолчанию |
|---|---|---|
| Макс. размер одного файла | `MAX_FILE_SIZE_MB` | 25 МБ |
| Макс. число файлов в задаче (scope=task) | `MAX_FILES_PER_TASK` | 20 |
| Макс. число файлов в отчёте (scope=report) | `MAX_FILES_PER_REPORT` | 10 |
| Макс. суммарный размер задачи | `MAX_TASK_TOTAL_MB` | 200 МБ |
| Антивирус-хук | `ANTIVIRUS_CMD` (опц.) | — (выключен) |

- **Обязательная валидация:** непустое имя файла (`UNSUPPORTED_FILENAME` иначе), размер ≤ лимита (потоковая проверка с обрывом загрузки при превышении → `413`, чтобы не принимать гигантский поток в память), число файлов ≤ лимита (`409 ATTACHMENTS_LIMIT`). Имя файла **санитизируется** для заголовка `Content-Disposition` (RFC 5987, `filename*=UTF-8''…`), на диск не попадает.
- **Антивирус-хук — опционально (раздел 10):** если задан `ANTIVIRUS_CMD` (например, `clamdscan`), файл сохраняется во временный путь, сканируется, и лишь при чистом результате перемещается в постоянный каталог; при срабатывании — `422 VALIDATION_ERROR` с `code: "FILE_REJECTED_AV"`. Реализуется как обёртка над `StorageBackend.save` (адаптер), чтобы не размазывать логику.
- **Ссылки (kind=link)** не пишутся на диск — это запись в БД с `url`; раздаются как обычное поле (без проксирования). Валидируется схема `http(s)://` и длина.

**Безопасная раздача (раздел 10, ключевое требование).** Файлы **никогда** не отдаются как статика по предсказуемому публичному URL. Единственный путь скачивания — `GET …/attachments/{att_id}/download`, который:
1. требует валидный Bearer-токен и непрекращённый доступ к реестру;
2. проверяет, что задача ∈ `visible_tasks(user)` (иначе `404`);
3. отдаёт поток через `StreamingResponse` напрямую из приватного тома (том **не** смонтирован во фронтенд/`nginx static`).

**Проектное решение (защищённая раздача).** Для FastAPI выбрана прямая отдача через приложение (а не `X-Accel-Redirect`/`X-Sendfile` от nginx), потому что школьная нагрузка низкая, а связка с проверкой прав — проще и надёжнее. Если позже понадобится разгрузить backend — `download` можно перевести на выдачу **подписанной короткоживущей ссылки** (HMAC + TTL) и отдачу через nginx `internal location`; интерфейс эндпойнта при этом не меняется. На согласование с заказчиком не выносится (внутренняя оптимизация).

### 13.6. Архитектура фронтенда (React + TypeScript)

Зона охватывает клиентское SPA: сборку, маршрутизацию, разделение серверного и клиентского состояния, API-клиент с авторизацией, структуру экранов (три ипостаси, карточка задачи, поиск, админ-панель), таблицы с сортировкой, печатную выгрузку и адаптивность. Серверная модель данных, контракт REST-эндпойнтов и планировщик уведомлений проектируются в смежных подразделах (см. 13.2 «Модель данных», 13.5 «REST API», 13.4 «Планировщик»); здесь фронтенд **потребляет** их контракт и нигде не дублирует серверную логику прав — клиентские проверки роли носят характер UX-подсказок, авторитетной остаётся серверная проверка (раздел 6, 11 ТЗ).

#### 13.6.1. Технологический выбор и обоснование

| Решение | Выбор | Обоснование под школьный self-host |
|---|---|---|
| Сборщик/дев-сервер | **Vite** | Мгновенный HMR, нулевая конфигурация, статическая сборка `dist/` раздаётся nginx внутри одного контейнера. Не тянет тяжёлый тулчейн — важно для скромного сервера. |
| Язык | **TypeScript (strict)** | Контракт API типизируется один раз (см. §13.6.4), ошибки прав/ролей ловятся компилятором. |
| Маршрутизация | **react-router (v6, data router)** | Декларативные вложенные маршруты, `loader`/`ErrorBoundary` на уровне роутов, защищённые секции через layout-route. |
| Серверное состояние | **TanStack Query (React Query)** | Кэш, инвалидация, фоновое обновление, дедупликация запросов. Списки трёх ипостасей и карточка задачи — это серверное состояние, ему не место в Redux. |
| Клиентское состояние | **Zustand** (лёгкий стор) | Только эфемерное клиентское: активная ипостась, токены сессии (в памяти), состояние модалок/тостов, параметры сортировки. Никакого «глобального стора задач». |

| Дата/время | **date-fns** (+ `date-fns-tz`) | Лёгкая, tree-shakeable; режим «дата+время»/«только дата» и счётчик «дней до дедлайна» (раздел 5, 7 ТЗ). Таймзона отображения — `Europe/Moscow` (согласуется с 13.4 «Планировщик»). |

#### 13.6.2. Управление состоянием: серверное vs клиентское

Чёткое разделение во избежание дублирования источника правды:

- **Серверное состояние (TanStack Query)** — всё, что приходит из API: списки задач по ипостаси, карточка задачи, реестр почт, профиль текущего пользователя. Ключи кэша строятся фабрикой:

```ts
// src/shared/api/queryKeys.ts
export const qk = {
  me: ['me'] as const,
  tasks: (role: TaskRole, sort: SortState) => ['tasks', role, sort] as const,
  task: (id: string) => ['task', id] as const,
  search: (code: string) => ['tasks', 'search', code] as const,
  registry: ['admin', 'registry'] as const,
};
```

  Мутации (создать/изменить/удалить задачу, сменить статус, отчёт, отметка готовности) инвалидируют соответствующие ключи. Поскольку статус и срок задачи влияют на сортировку и на флаг «Просрочена», после `PATCH /tasks/{id}/status` и `PUT /tasks/{id}` инвалидируются и `qk.task(id)`, и все `['tasks', ...]`.

- **Клиентское состояние (Zustand)** — активная ипостась (`'author' | 'assignee' | 'observer'`), состояние сортировки каждой таблицы, открытая модалка-карточка, очередь тостов, in-memory access-токен. Стор не сериализует задачи.

> **Проектное решение (флаг «Просрочена» на клиенте).** Раздел 5 ТЗ требует достоверности флага независимо от планировщика. Сервер возвращает в DTO задачи как сам флаг `is_overdue`, так и `deadline` + `deadline_has_time`. Клиент **дополнительно** вычисляет «визуальную просрочку» по `now ≥ момент_срока` для случая, когда задача открыта в кэше дольше прохода планировщика. Авторитетен сервер; клиентский расчёт лишь не даёт показать «не просрочена» по устаревшему кэшу. Бейдж просрочки — отдельная пометка, не статус (раздел 5 ТЗ).

#### 13.6.3. API-клиент, хранение токена и авторизация

```ts
// src/shared/api/http.ts
import axios from 'axios';
import { useAuthStore } from '@/shared/auth/store';

export const http = axios.create({ baseURL: import.meta.env.VITE_API_BASE_URL });

http.interceptors.request.use((config) => {
  const token = useAuthStore.getState().accessToken;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

http.interceptors.response.use(
  (r) => r,
  async (error) => {
    if (error.response?.status === 401) {
      // удалён из реестра / истёкший токен → принудительный разлогин (раздел 3 п.11–12 ТЗ)
      useAuthStore.getState().logout();
      // навигация на /login выполняется роут-гардом по отсутствию сессии
    }
    return Promise.reject(error);
  },
);
```

**Хранение токена — проектное решение.** Внутренний инструмент за периметром школы; выбираем баланс простоты и безопасности:

- **Access-токен (короткоживущий JWT) хранится в памяти** (Zustand, не в `localStorage`) — минимизирует XSS-кражу.
- **Refresh-токен — в `HttpOnly`+`Secure`+`SameSite=Strict` cookie**, выставляется бэкендом; фронтенд его не читает. При старте приложения и при 401 — попытка тихого `POST /auth/refresh`; неудача → разлогин и редирект на `/login`.

> Если 13.5 «REST API» решит обойтись без refresh-механики (короткая сессия + повторный вход), фронтенд деградирует корректно: при 401 сразу разлогинивает. Это согласуется с разделом 3 п.12 ТЗ — серверная проверка доступа на каждый запрос, удалённый из реестра пользователь мгновенно теряет доступ.

**Обработка отказа в регистрации (раздел 3 п.3 ТЗ).** Эндпойнт `POST /auth/register` для e-mail вне реестра возвращает машинно-распознаваемый код ошибки; фронтенд показывает **ровно** фразу «Извините, у вас нет доступа к сервису.» из `strings.ts`, без каких-либо контактов, подсказок и переадресации (раздел 2, 3 ТЗ). Фраза — единственная константа, изменять её нельзя.

#### 13.6.7. Рендер карточки задачи по роли (UX-проекция матрицы прав)

`TaskCard` определяет роль текущего пользователя в конкретной задаче (`me.id` против `author/assignee/observers`) и рендерит соответствующий набор контролов. Это **только UX** — все мутации сервер перепроверяет (раздел 6, 11 ТЗ):

| Контрол | Постановщик | Исполнитель | Наблюдатель |
|---|---|---|---|
| Поля задачи (название/описание/срок/исполнитель/наблюдатели/вложения) | редактирование | только просмотр | только просмотр |
| Смена статуса (`StatusControl`) | да | скрыт | скрыт |
| Удалить задачу | да (+ подтверждение) | скрыт | скрыт |

| «Отписаться о готовности» (`ReadyButton`) | скрыт | да | скрыт |

Наблюдатель видит карточку только если задача в его списке (раздел 4 ТЗ); попытка открыть `/tasks/:id` вне прав → сервер вернёт 403/404, фронтенд покажет `EmptyState` «Задача недоступна».

#### 13.6.8. Таблицы и сортировка

Все три ипостаси используют один `TasksTable` с конфигом колонок. Колонки: № (seqNo), ID (code), Название, Срок, Статус, бейдж «Просрочена», контекстное лицо (исполнитель/постановщик/наблюдатели по вкладке), а во вкладке постановщика — `ReadyIndicator` «Готовность от исполнителя» (раздел 8 п.1 ТЗ). Клик по любому заголовку сортирует по этому полю (раздел 8 п.2).

```ts
// сортировка по умолчанию (src/features/tasks-table/useTableSort.ts)
const isOpen = (t: TaskListItem) => t.status === 'in_progress';
const defaultComparator = (a, b) =>
  Number(isOpen(b)) - Number(isOpen(a))   // вторичный ключ: открытые выше закрытых
  || a.deadline.localeCompare(b.deadline); // первичный смысл: ближайший срок выше
```

> **Проектное решение (вторичный ключ сортировки — раздел 8 п.3 ТЗ).** По умолчанию сортируем по сроку (ближайший выше — прямое требование директора), но первым неявным ключом ставим «открытые (`В работе`) выше закрытых (`Выполнена`/`Отменена`)». Директором явно не закреплено — **вынести на согласование**. Параметр сортировки уходит и на сервер (для серверной пагинации в будущем), и применяется на клиенте для текущей выборки.

#### 13.6.9. Печатная выгрузка (print-CSS)

Раздел 8 п.5 ТЗ: «отформатировано под печать, пригодно к пересылке», простое решение для self-host — **печатная HTML-страница, без генерации PDF на клиенте**.

- Маршрут `/print` рендерит `PrintView` тем же отфильтрованным/отсортированным списком, что и активная таблица (параметры передаются через query/`location.state`).
- `@media print` в отдельном `print.module.css`: скрывает сайдбар/кнопки/тосты (`display:none`), задаёт `@page { size: A4; margin: 12mm }`, печатает читаемую таблицу со всеми полями (номер, ID, название, описание/срок, исполнитель, постановщик, наблюдатели, статус, флаг просрочки, отчёт), управляет разрывами строк (`break-inside: avoid` для строк, `thead { display: table-header-group }` — повтор шапки на каждой странице).
- Кнопка «Печать/Сохранить» вызывает `window.print()`; пользователь сохраняет в PDF средствами браузера и пересылает. PDF-генерация на сервере — опциональна (см. 13.5 «REST API», эндпойнт `export?format=print`).

```css
@media print {
  .appShell__sidebar, .appShell__actions, .toastHost { display: none; }
  @page { size: A4; margin: 12mm; }
  .printTable tr { break-inside: avoid; }
  .printTable thead { display: table-header-group; }
}
```

#### 13.6.10. Адаптивность и доступность

- **Брейкпоинты (CSS-переменные/медиазапросы):** `≥1024px` — сайдбар слева + таблица; `768–1023px` — сворачиваемый сайдбар; `<768px` — `MobileTopTabs` (верхние вкладки/бургер), таблицы превращаются в карточный список строк (каждая задача — карточка с подписанными полями), чтобы корректно читалось с телефона (раздел 7 п.6, 10 ТЗ).
- **Доступность:** семантические `<table>`/`<th scope>`, `aria-sort` на сортируемых заголовках, фокус-ловушка в модалке-карточке и в `ConfirmDialog`, управление с клавиатуры, видимый фокус, контрастные бейджи (просрочка не только цветом, но и текстом «Просрочена»).
- **Локализация:** только русский, формат дат `dd.MM.yyyy` / `dd.MM.yyyy HH:mm` (date-fns, locale `ru`).

#### 13.6.11. Сборка и развёртывание

Vite собирает статику в `dist/`, которая обслуживается nginx во frontend-контейнере `docker-compose` (раздел 10 ТЗ); nginx также проксирует `/api` на backend и `/uploads` (или отдаёт вложения), что снимает CORS. `VITE_API_BASE_URL` и базовый URL берутся из переменных окружения сборки (`.env.example`), что закрывает требование развёртывания на школьном сервере с динамическим IP — адрес задаётся при деплое, а не вшит в код.

### 13.7. Тестирование, миграции, сид и развёртывание

Зона отвечает за то, чтобы инвариантами из разделов 3–11 можно было **доказательно** управлять: автотесты фиксируют поведение, миграции — структуру, сид — стартовое состояние, а `docker compose up` — воспроизводимый запуск. Ссылки на сущности БД и типы событий планировщика — по подразделам зоны «Модель данных» (13.2) и «Уведомления/планировщик» (13.4); здесь мы их **проверяем**, а не переопределяем.

#### 13.7.1. Стратегия автотестов (пирамида)

Используем классическую пирамиду с акцентом на средний слой — большинство критичных правил системы это правила **на стыке БД и API**, а не чистая арифметика, поэтому интеграционных тестов будет больше, чем юнитов.

```
            ┌───────────────────────────┐
            │  e2e (smoke, опционально) │  ← 3–5 сценариев через TestClient+UI? нет:
            │  только happy-path API    │     только API happy-path (login→create→report→close)
            ├───────────────────────────┤
            │   Интеграционные (API +   │  ← ОСНОВНОЙ слой: матрица прав, нумерация,
            │   реальная тест-БД)       │     ID-коллизии, удаление архива, поиск
            ├───────────────────────────┤
            │   Юнит (доменные правила) │  ← чистые функции: момент просрочки,
            │   без БД и без сети       │     получатели/тайминг уведомлений, валидаторы
            └───────────────────────────┘
```

| Слой | Что покрывает | Инструмент | БД | Время |
|---|---|---|---|---|
| **Юнит** | `is_overdue(deadline, mode, now)`, расчёт получателей и дня события (1–4), генератор 6-значного кода, валидаторы (1 исполнитель, ≤5 наблюдателей, обязательные поля) | `pytest` | нет | `freezegun` |
| **Интеграционные** | все REST-эндпойнты (10.1) с проверкой ролей, сквозная нумерация, уникальность ID, удаление пользователя+архива, прогон планировщика и журнал идемпотентности | `pytest` + `httpx.AsyncClient`/`TestClient` | тест-PostgreSQL | `freezegun` + фейк-SMTP/MAX |
| **e2e smoke** | один сквозной путь «регистрация → постановка → отчёт → закрытие» | `pytest` (API-уровень) | тест-PostgreSQL | — |

**Проектное решение.** Frontend (React) покрываем отдельно лёгкими компонентными тестами (Vitest + React Testing Library) только на критичную логику отображения (бейдж «Просрочена», скрытие кнопок по роли, текст отказа). E2e через браузер (Playwright) — **за рамками MVP**: серверная проверка прав — основной рубеж (разд. 11, 6), а UI-блокировки лишь дублируют её; полноценный браузерный e2e раздувает объём. Вынести на подтверждение заказчику.

**Целевое покрытие.** Не гонимся за процентом строк; обязательны к покрытию **все** ячейки матрицы прав (включая негативные) и **все** перечисленные в разд. 11 бизнес-правила. CI считает покрытие (`pytest --cov`) информативно, порог — мягкий (например, `--cov-fail-under=80` на пакете `domain/` и `api/`).

#### 13.7.2. Инфраструктура тестов и фикстуры

**Тест-БД — настоящий PostgreSQL, не SQLite.** Проект завязан на серверную нумерацию, уникальные индексы, `ON DELETE CASCADE` и (возможно) advisory-locks — поведение SQLite здесь расходится. БД поднимается контейнером для CI и локально (`docker compose -f docker-compose.test.yml`), либо адресуется через `TEST_DATABASE_URL`.

**Изоляция — транзакция-обёртка с откатом (`transactional rollback`).** Схема создаётся один раз на сессию через миграции Alembic (см. 13.7.5), затем каждый тест выполняется во вложенной транзакции, которая откатывается на teardown. Это даёт быстрые тесты без пересоздания схемы.

```python
# conftest.py (фрагмент)
@pytest.fixture(scope="session")
def engine():
    eng = create_engine(settings.TEST_DATABASE_URL, future=True)
    run_alembic_upgrade_head(eng)          # схема = прод-миграции, не metadata.create_all
    yield eng
    eng.dispose()

@pytest.fixture
def db(engine):
    conn = engine.connect()
    outer = conn.begin()                   # внешняя транзакция
    session = Session(bind=conn, join_transaction_mode="create_savepoint")
    yield session                          # тест видит чистую БД
    session.close()
    outer.rollback()                       # полный откат всего, что сделал тест
    conn.close()

@pytest.fixture
def client(db):                            # переопределяем зависимость get_db на тестовую сессию
    app.dependency_overrides[get_db] = lambda: db
    return TestClient(app)
```

**Важное следствие для нумерации.** Сквозной счётчик (см. 13.2) при откате транзакции тоже откатывается, поэтому тесты на «отсутствие дыр» должны идти **внутри одной** транзакции теста — иначе откат предыдущего теста создаст ложную «дыру». Это поведение мы делаем явным предметом проверки (см. 13.7.3, п. «нумерация»).

**Заморозка времени — `freezegun`.** Все тесты просрочки, дедлайнов и графика уведомлений запускаются под `freeze_time`, время двигаем явно (`tick`/повторный `freeze_time`). Серверный код **обязан** брать «сейчас» из единой инъецируемой функции `now()` (а не `datetime.now()` вразнобой), иначе `freezegun` не накроет фоновые модули. Это требование к коду планировщика (13.4).

**Фейковые внешние каналы.** SMTP и MAX за интерфейсом канала (адаптеры из 13.4) подменяются in-memory реализациями, которые копят отправленное в список:

```python
class FakeChannel:
    def __init__(self): self.sent = []     # [(channel, to, event_type, task_no, day)]
    async def send(self, msg): self.sent.append(msg.key()); return True

@pytest.fixture
def outbox(monkeypatch):
    fake = FakeChannel()
    monkeypatch.setattr(notifications, "EMAIL", fake)
    monkeypatch.setattr(notifications, "MAX", fake)
    return fake
```

Никаких реальных SMTP-соединений и сетевых вызовов MAX в тестах. Адаптер MAX в тестах настраивается на сценарии «привязка есть» / «привязки нет» / «канал недоступен (best-effort упал)» — проверяем, что падение MAX не блокирует e-mail (разд. 9).

**Фабрики данных.** Лёгкие хелперы `make_user(email=...)`, `make_task(author=, assignee=, observers=, deadline=, mode=)` поверх сессии (например, через `factory_boy` либо простые функции) — чтобы тесты читались как сценарии, а не как наполнение таблиц.

#### 13.7.3. Обязательные тест-наборы (трассировка к разд. 11)

Ниже — перечень с пометкой, какой инвариант ТЗ закрывает каждый набор. Это контракт зоны: набор считается готовым, когда покрыты **все** строки.

**А. Матрица прав (разд. 6) — позитив и негатив.** Для каждой ячейки таблицы прав — отдельный тест; негативные кейсы возвращают `403` и **не меняют состояние** (после запрета повторное чтение подтверждает неизменность поля).

| Кейс (негативный приоритетно) | Ожидание |
|---|---|
| Исполнитель `PUT /tasks/{id}` (любое поле) | `403`, поля без изменений |
| Исполнитель `PATCH /tasks/{id}/status` | `403`, статус прежний |
| Исполнитель `DELETE /tasks/{id}` | `403` |
| Наблюдатель любой мутирующий вызов | `403` |
| Наблюдатель `GET` чужой задачи (где он не наблюдатель) | `404` (а не `403` — не раскрываем существование; см. область видимости поиска, 7.5) |
| Наблюдатель `GET` «своей» задачи | `200`, видит поля + отчёт |
| Постановщик `POST /tasks/{id}/report` | `403` (отчёт пишет только исполнитель) |
| Не-исполнитель `POST /tasks/{id}/mark-ready` | `403` |
| Постановщик — полный CRUD своей задачи | `200/204` |
| Поиск `GET /tasks/search?code=` за пределами видимости | `404` (поиск ограничен правами, 7.5) |

**Проектное решение (для видимости/поиска):** на запрос к задаче вне прав возвращаем `404`, а не `403`, чтобы не подтверждать существование чужой задачи постороннему — согласуется с приватностью наблюдателя (разд. 4) и областью поиска (7.5). Зафиксировать в зоне API (13.5); тест закрепляет именно `404`.

**Б. Сквозная нумерация (разд. 5).**
- Создание N задач подряд → номера строго `k, k+1, … k+N−1`, без дыр и дублей (проверка внутри одной транзакции теста).
- Параллельное создание (две конкурентные транзакции) → нет двух задач с одним номером (проверяем механизм из 13.2: `BIGSERIAL`/секвенция или `SELECT … FOR UPDATE` на счётчике). **Проектное решение:** тест на гонку — через два соединения в одном тесте; если в 13.2 выбран `BIGSERIAL`, тест подтверждает монотонность; если выбран счётчик-строка — подтверждает блокировку.
- Откат создания (ошибка валидации) и поведение номера документируется тестом (для `BIGSERIAL` номер «прожигается» — это допустимо для сквозного **номера**; «без дыр» в строгом смысле гарантируется только для счётчика-строки). **Открытый вопрос вынесен в 13.2.**

**В. Уникальность 6-значного ID (разд. 5).**
- Сгенерированный код всегда в `100000–999999`.
- Коллизия: мокаем генератор так, чтобы первые попытки вернули уже занятый код → код перегенерируется, запись сохраняется, итог уникален; уникальный индекс БД ловит гонку (вставка дубля → `IntegrityError` → ретрай).
- Ограничение числа ретраев: после разумного лимита (например, 10) — внятная ошибка, а не вечный цикл (важно при близком заполнении диапазона — теоретический край, тест документирует поведение).

**Г. Флаг «Просрочена» (разд. 5).**
- Флаг — авто и **не меняет статус**: задача `В работе` + `Просрочена` одновременно (проверяем оба поля).
- Режим «дата+время»: при `now < момент` — не просрочена; при `now ≥ момент` — просрочена (граница `==` включительно).
- Режим «только дата»: в конце дня-срока **ещё не** просрочена; с началом следующих суток — просрочена (граница на полуночь следующего дня в таймзоне организации).
- Достоверность «по факту»: флаг корректен при чтении даже без прогона суточного воркера (вычисление `now ≥ момент_срока` / частый лёгкий проход — см. 13.4).
- «Факт не обнуляется»: после того как флаг возник, он остаётся в отчётности даже после закрытия задачи (прекращаются лишь ежедневные напоминания).
- Изменение срока постановщиком: новый срок в будущем → флаг снимается (переоценка, разд. 9); новый срок в прошлом → флаг остаётся/выставляется.

**Д. График 4 уведомлений (разд. 9) — получатели, тайминг, краевые, идемпотентность.**

| Проверка | Ожидаемое |
|---|---|
| Событие №1 (день постановки) | получают **исполнитель И все наблюдатели**, разный текст; за пределами — никто |
| Событие №2 (за сутки) | **только исполнитель** |
| Событие №3 (день срока) | **только исполнитель** |
| Событие №4 (после срока, ежедневно) | **только исполнитель**, каждый день пока не закрыта |
| Наблюдатель | получает **только №1** и больше ничего |
| Краевой: задача создана в день срока | №1 уходит, №2 пропускается, №3 уходит |
| Краевой: задача создана после срока | №1 уходит, сразу флаг «Просрочена», далее №4 |
| Идемпотентность | повторный прогон воркера за тот же день — `outbox` без новых записей (журнал по `task+тип+день`) |
| Перезапуск воркера в середине дня | дублей нет |
| Закрытие задачи (`Выполнена`/`Отменена`) | №4 прекращается со следующего прогона |
| Смена срока | ключи идемпотентности привязаны к версии срока → №2/№3/№4 срабатывают заново по новому сроку |
| MAX недоступен | e-mail всё равно уходит; журнал помечает e-mail как доставленный (best-effort MAX не ломает обязательный канал) |

Тайминг во всех кейсах двигаем `freeze_time`; «сутки» вычисляем по правилам режима срока (разд. 9). Идемпотентность проверяем именно через **журнал отправленных уведомлений** (13.4/разд. 11), а не косвенно.

**Е. Удаление пользователя (разд. 3, п. 10–11).**
- `DELETE /admin/users/{id}` (только админ; не-админ → `403`) удаляет все задачи, где пользователь **постановщик**, и его вложения/отчёты (каскад из 13.2).
- После удаления вход блокируется: `POST /auth/login` → отказ; токен/сессия удалённого не проходит серверную проверку на следующем запросе (разд. 3, п. 11–12).
- Следы в **чужих** задачах (был исполнителем/наблюдателем) — поведение по выбранному в 13.2 решению (по умолчанию «пометить участие как удалён»); тест закрепляет именно согласованный вариант. **Открытый вопрос (объём «архива») — за заказчиком, разд. 3 п. 10.1.**

**Ж. Отказ в регистрации вне реестра (разд. 3).**
- `POST /auth/register` с e-mail вне реестра → тело ответа содержит **ровно** строку `«Извините, у вас нет доступа к сервису.»` (точное сравнение, включая регистр и точку), без контактов/подсказок.
- E-mail в реестре → регистрация проходит, задаётся пароль.
- Тест на точность фразы выносим в **константу** и сравниваем с ней и в backend, и (для фронта) в компонентном тесте — чтобы фраза не «разъехалась».

**З. Серверная валидация (разд. 11).** Ровно один исполнитель (0 или 2 → `422`), ≤5 наблюдателей (6 → `422`), обязательные название и срок (пусто → `422`), исполнитель ≠ дубль в наблюдателях (если решено в 13.5) — тестируется на `POST /tasks` и `PUT /tasks/{id}`.

#### 13.7.4. Безопасность (рубеж — сервер)

Серверная проверка прав — **основной и единственно надёжный** рубеж (разд. 6, 11); UI лишь прячет кнопки. Меры зоны:

| Мера | Решение |
|---|---|
| Хеш паролей | **argon2id** (через `passlib`/`argon2-cffi`), параметры вынесены в конфиг; **bcrypt** — допустимая запасная опция. Никогда не храним и не логируем пароли. |
| Сброс пароля | одноразовый токен с TTL (разд. 3, п. 6), хранится хешем, инвалидируется после использования; тест на истёкший/повторно использованный токен |
| Лимиты загрузки | макс. размер файла и число вложений из env (`MAX_UPLOAD_MB`, `MAX_ATTACHMENTS`); серверная проверка размера и имени (санитизация имени, запрет path-traversal); тип не ограничиваем (разд. 5), но размер/имя — обязательно (разд. 10) |
| Хранение вложений | файловое, вне webroot; отдача только через эндпойнт с проверкой прав (нельзя получить чужой файл по прямой ссылке) |
| CORS | разрешённые origin из env (`CORS_ORIGINS`), по умолчанию только `BASE_URL`; не `*` при включённых cook*/credentials |
| Security-заголовки | `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy`, `Content-Security-Policy` (узкий), `Strict-Transport-Security` при HTTPS — через middleware |
| Секреты | только через env/`.env` (не в репозитории), `JWT_SECRET`/`SECRET_KEY` обязательны и без дефолтов в проде; `.env` в `.gitignore`, в репозитории — только `.env.example` |
| Rate-limit на auth | базовый лимит на `/auth/login` и `/auth/password-reset/request` (защита от перебора) — простой in-memory/БД-счётчик, без внешних зависимостей |
| Сессия удалённого | проверка реестра/активности на **каждый** запрос (разд. 3, п. 12) |

Безопасность покрывается тестами там, где это инвариант (отказ удалённому, `403` без прав, чужой файл недоступен, истёкший reset-токен), а конфигурацию заголовков/CORS проверяем одним smoke-тестом ответа.

#### 13.7.5. Миграции (Alembic)

- **Единственный источник схемы — Alembic.** В тестах схема создаётся прогоном `upgrade head` (а не `metadata.create_all`), чтобы тестировалась та же схема, что в проде, включая ручные правки миграций (частичные/функциональные индексы, `CHECK`-констрейнты на число наблюдателей и т.п.).
- Все ограничения-инварианты по возможности **на уровне БД**: уникальный индекс на 6-значный код, уникальность сквозного номера, `FK ON DELETE CASCADE` для архива пользователя, `CHECK (observers_count <= 5)` где применимо, частичный уникальный индекс для активной привязки канала. Это страхует прикладную валидацию.
- Тест миграций: `upgrade head` → `downgrade base` → `upgrade head` проходит без ошибок (миграции обратимы и согласованы); отдельный CI-шаг.
- Контейнер `backend` при старте выполняет `alembic upgrade head` (через entrypoint) до запуска API — БД всегда на актуальной версии.

#### 13.7.6. Сид-скрипт

Идемпотентный скрипт `python -m app.seed` (вызывается из entrypoint при `RUN_SEED=true` или вручную):

1. **Первичный администратор** из env: `ADMIN_EMAIL`, `ADMIN_PASSWORD`. Скрипт вносит e-mail в реестр и создаёт учётную запись с флагом администратора. Если админ уже есть — пропуск (идемпотентность), при пустых env — внятная ошибка.
2. **Начальный реестр e-mail** — из файла (`SEED_REGISTRY_FILE`, например CSV/TXT со списком сотрудников) либо пусто; повторный запуск не плодит дублей (`INSERT … ON CONFLICT DO NOTHING` по e-mail).
3. Скрипт **не** создаёт демо-задачи в проде (только опционально под флагом `SEED_DEMO=true` для дев-стенда).

```bash
# .env (фрагмент)
ADMIN_EMAIL=director@school2090.ru
ADMIN_PASSWORD=...            # задаётся при развёртывании, не в репозитории
SEED_REGISTRY_FILE=./seed/registry.txt
RUN_SEED=true
```

Тест сид-скрипта: на чистой тест-БД `seed` создаёт ровно одного администратора и заявленные записи реестра; повторный запуск не меняет состояние (идемпотентность).

#### 13.7.7. Развёртывание (`docker compose up` одной командой)

Состав compose (один файл, разд. 10):

```
┌──────────────────────────────────────────────────────────┐
│ docker-compose.yml                                        │
│                                                           │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌────────┐ │
│  │ frontend │   │ backend  │   │  worker  │   │   db   │ │
│  │ (nginx,  │──▶│ FastAPI  │──▶│ cron-loop│──▶│Postgres│ │
│  │  React   │   │ uvicorn  │   │(тот же   │   │  15    │ │
│  │  build)  │   │          │   │ образ)   │   │        │ │
│  └──────────┘   └────┬─────┘   └────┬─────┘   └───┬────┘ │
│                      │              │             │      │
│                      └─────── volume: attachments ┘      │
│                      └─────── volume: pgdata ────────────┘
└──────────────────────────────────────────────────────────┘
```

- **`db`** — `postgres:15`, данные на volume `pgdata`, healthcheck (`pg_isready`).
- **`backend`** — FastAPI/uvicorn; entrypoint: ждёт `db` healthy → `alembic upgrade head` → (опц.) `seed` → запуск API. `depends_on: db (healthy)`.
- **`worker`** — **тот же образ**, что backend, но команда — встроенный cron-loop (без внешнего брокера, разд. 10): лёгкий частый проход для флага «Просрочена» (5–15 мин) + суточная рассылка в `NOTIFY_HOUR`/`TZ` (см. 13.4). Делит код и БД с backend.
- **`frontend`** — статическая сборка React, раздаётся nginx; проксирует `/api` на backend; знает `BASE_URL` на этапе сборки/рантайма.
- **Тома:** `pgdata` (БД), `attachments` (файловое хранилище вложений, разд. 5/10) — оба персистентны.

**`.env.example`** содержит все переменные (разд. 10): `DATABASE_URL`, `SMTP_*`, `MAX_*`, `JWT_SECRET`/`SECRET_KEY`, `ATTACHMENTS_DIR`, `MAX_UPLOAD_MB`, `MAX_ATTACHMENTS`, `TZ=Europe/Moscow`, `NOTIFY_HOUR=09:00`, `BASE_URL`, `CORS_ORIGINS`, `ADMIN_EMAIL`, `ADMIN_PASSWORD`, `SEED_REGISTRY_FILE`. Реальный `.env` — в `.gitignore`.

**Динамический IP школьного сервера (разд. 10) — в README, три варианта по возрастанию усилий:**

| Вариант | Когда | Суть |
|---|---|---|
| 1. Статический IP у провайдера | проще всего, если провайдер даёт | заказать статический IP, прокинуть порт, `BASE_URL` = `https://<домен или IP>` |
| 2. VPS со статическим IP | школьный сервер недоступен извне | развернуть тот же compose на дешёвом VPS, домен на него |
| 3. Reverse-proxy / туннель | нет белого IP и нет VPS | туннель (напр. Cloudflare Tunnel/аналог) с публичного домена на локальный сервер; `BASE_URL` = публичный домен |

Во всех вариантах **`BASE_URL` задаётся через env** и используется в письмах (ссылки сброса пароля, ссылки на задачи) и в CORS — это единственная точка правки при смене адреса.

**README (на русском, разд. 10)** включает: назначение; требования (Docker + Compose); быстрый старт (`cp .env.example .env` → заполнить → `docker compose up -d`); таблицу переменных; создание первичного администратора (env + сид); настройку SMTP и MAX; раздел про динамический IP (таблица выше); лицензию (открытая/бесплатная, для self-host с GitHub).

#### 13.7.8. CI (минимум)

GitHub Actions: на push/PR — поднять `postgres` service-контейнер, `alembic upgrade head`, `pytest` (юнит+интеграционные) с `freezegun` и фейк-каналами, прогон `down/up` миграций, линт (`ruff`) и типизация (`mypy`) backend, `vitest` для критичных компонентов фронта. Зелёный CI — условие готовности зоны.

Ниже собраны решения, помеченные по тексту раздела 13 как **«Проектное решение»** и требующие подтверждения заказчика (перекликается с разделом 12, п. 1). До подтверждения реализуется вариант «по умолчанию», указанный рядом.

**Данные и удаление пользователя (разделы 3, 5):**
- **Объём «архива» при удалении пользователя** (раздел 3, п. 10.1): по умолчанию — каскадно удалять задачи, где он **постановщик**, а в чужих задачах участие **обезличивать** («Пользователь удалён»). На выбор: снять / обезличить / пометить. См. 13.2.9, 13.3.3, 13.5.4.
- **Судьба чужой задачи, где удаляемый был единственным исполнителем** (исполнитель обязателен, раздел 4): варианты — (а) система переводит задачу в `Отменена`; (б) постановщик обязан переназначить (`needs_reassignment`); (в) «технический» псевдо-исполнитель. По умолчанию выносится на решение директора. См. 13.5.4.
- **Стратегия сквозного номера без дыр**: таблица-счётчик с блокировкой строки (строго без дыр) vs `SEQUENCE` (возможны дыры при откате). По умолчанию — таблица-счётчик. См. 13.2.5.

**Сроки, просрочка, уведомления (разделы 5, 9):**
- **Момент просрочки для режима «только дата»**: с началом следующих суток после даты срока (в `TZ` организации). См. 13.2 (DDL `tasks`), 13.4.2.
- **Время суточной рассылки и таймзона**: по умолчанию `NOTIFY_TIME=09:00`, `TZ=Europe/Moscow`. См. 13.4.6, 13.1.6.
- **Интервал частого прохода флага просрочки**: `OVERDUE_SWEEP_MINUTES=10` (5–15 мин). См. 13.4.6.

**Списки, поиск, выгрузка (разделы 7, 8):**
- **Вторичный ключ сортировки** «открытые (`В работе`) выше закрытых, затем по сроку» — рекомендация разработчика, директором явно не закреплена. См. 13.5.4, 13.6.8.
- **Область видимости поиска по 6-значному ID** — строго в пределах прав роли; для скрытых ресурсов возвращается `404`, а не `403`. См. 13.3.6, 13.5.2–13.5.3.
- **Формат выгрузки под печать**: по умолчанию print-HTML; PDF — опционально (флаг окружения). См. 13.5.5, 13.6.9.

**Аутентификация и контракт (разделы 3, 10):**
- **Схема сессии**: JWT (короткий access + refresh в httpOnly-cookie, refresh в БД) vs серверные сессии. По умолчанию — JWT-гибрид; TTL access по умолчанию 15 мин. См. 13.3.2.
- **Способ задания первичного администратора**: прямой `ADMIN_PASSWORD` (по ТЗ) vs одноразовый `ADMIN_INITIAL_TOKEN` (безопаснее в проде). По умолчанию поддержаны оба. См. 13.1.6, 13.3.5.
- **Версионирование API** (`/api/v1`) и UUID задачи в путях (а не номер/код). См. 13.5.1.
- **Разделение операций**: `DELETE /admin/registry/{id}` (мягко лишить доступа) и `DELETE /admin/users/{id}` (жёсткое удаление архива) — два разных эндпойнта. См. 13.5.4.
- **Запрет «исполнитель = наблюдатель»** в одной задаче — на согласование. См. 13.3.7, 13.5.4.

**Инфраструктура и тесты (разделы 10, 11):**
- **Reverse-proxy/TLS — Caddy** (автоматический HTTPS) с возможностью отключения (`--profile noproxy`), если у школы свой прокси/туннель. См. 13.1.1.
- **Расположение `migrations/` и `seeds/`** в корне репозитория (заметность для оператора) vs внутри `backend/`. См. 13.1.5.
- **Браузерный e2e (Playwright)** — за рамками MVP (серверная проверка прав — основной рубеж); компонентные тесты фронта — да. См. 13.7.1.
